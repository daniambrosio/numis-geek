## Overview

Hoje a única entrada pra **criar uma opção** (compound: novo Asset
OPTION + movimento `SELL_OPEN`/`BUY_TO_OPEN`) é navegar até o
underlying, abrir o AssetDetail e usar `+ Nova opção` no
`OpenOptionsCard`. Pro use case real "fim de semana, abrir 3-5
PUTs em sequência", isso é 5+ cliques por opção e quebra o fluxo.

Os **lifecycle closers** (`SELL_TO_CLOSE`, `BUY_TO_CLOSE`,
`EXERCISED`, `EXPIRED`) também ficam órfãos: só acessíveis via
botões inline no `OpenOptionsCard`. Não dá pra lançar um close de
opção a partir de `/lançamentos`.

Esta spec resolve os dois buracos com mudanças cirúrgicas, sem
criar novo design surface:

1. **Adicionar "Opção" no `NovoButton`** (8º item, grupo
   Investimentos). Vira entrada global.
2. **Tornar o `OptionModal` standalone** — aceitar `underlying`
   opcional; quando ausente, primeiro campo é asset picker
   (`STOCK/REIT/ETF`).
3. **`MovementComposer` dinâmico pra OPTION** — quando o asset
   selecionado é uma opção, o type picker mostra
   `SELL_TO_CLOSE / BUY_TO_CLOSE / EXERCISED / EXPIRED` em vez
   do grid 3×2 padrão.
4. **"Salvar e abrir outra"** no `OptionModal` — botão extra pro
   use case batch.

Estabelece também o **padrão "compound create"**: cada classe de
ativo complexa (OPTION agora, FIXED_INCOME depois) ganha seu
próprio composer dedicado e um item no `NovoButton`. **Não**
unificar todos num composer genérico — campos são distintos
demais.

### 2. `OptionModal` standalone

`frontend/src/components/OptionModal.tsx` precisa de 3 mudanças:

**2.1 Props change**

```typescript
interface Props {
  underlying?: AssetOut         // ← era obrigatório, vira opcional
  onClose: () => void
  onSaved: (createdAsset: AssetOut) => void  // ← retorna o asset criado pro "abrir outra"
}
```

**2.2 Novo campo: Underlying picker**

Quando `underlying` é `undefined` no mount, renderizar como
primeiro campo:

```
┌─ Underlying ─────────────────────────────┐
│ [▾ Selecione…                         ▼] │
│   Filtrado: klass IN (STOCK, REIT, ETF)  │
│   Sort: ticker A-Z                       │
│   Mostra: ticker · name · price atual    │
└──────────────────────────────────────────┘
```

Componente: reutilizar `<AssetSelect />` se existir; senão criar.
Filter por `klass IN ('STOCK', 'REIT', 'ETF')` — não permitir
opção sobre opção, sobre crypto, sobre FGTS etc.

Quando user seleciona, faz internamente o mesmo trabalho que hoje
faz com o prop `underlying` pré-preenchido (passa para parser,
herda `account_id`).

Quando `underlying` vem por prop (caso AssetDetail atual), o
field **não renderiza** — comportamento idêntico ao de hoje.

**2.3 "Salvar e abrir outra"**

Segundo botão de save, ao lado do "Salvar" atual:

```
[Cancelar]                 [Salvar e abrir outra]  [Salvar]
```

Comportamento "Salvar e abrir outra":
1. POST `/options` com payload atual.
2. Em sucesso, **não** chama `onClose()`.
3. Limpa todos os campos exceto: `underlying` (mantém),
   `optionType` (mantém — usuário geralmente vende várias PUTs
   seguidas, não alterna).
4. Foca de volta no campo `ticker`.
5. Mostra toast "Opção {ticker} criada · prêmio R$ {x}".

Comportamento "Salvar" (existente): igual hoje, fecha modal.

### 3. `MovementComposer` dinâmico pra OPTION

`frontend/src/components/MovementComposer.tsx` — type picker
condicional ao asset selecionado.

**3.1 Detecção**

```typescript
const isOptionAsset = a?.asset_class === "OPTION"
```

**3.2 Type picker condicional**

Hoje o picker mostra `Object.entries(MV_TYPE_CFG).map(...)` em
grid 3×2. Mudança:

```typescript
const availableTypes = isOptionAsset
  ? OPTION_LIFECYCLE_TYPES  // SELL_TO_CLOSE, BUY_TO_CLOSE, EXERCISED, EXPIRED
  : Object.entries(MV_TYPE_CFG)  // BUY, SELL, BONUS, SUBSCRIPTION, COME_COTAS, FULL_REDEMPTION
```

Renderizar em grid 2×2 quando OPTION (4 tipos), 3×2 caso
contrário (6 tipos).

**3.3 `OPTION_LIFECYCLE_TYPES` — novo config**

```typescript
export const OPTION_LIFECYCLE_TYPES = {
  SELL_TO_CLOSE: {
    label: "Vender / Encerrar",
    hint: "Recompra a opção vendida",
    qty: true, price: true, fee: true, tax: false,
  },
  BUY_TO_CLOSE: {
    label: "Comprar / Encerrar",
    hint: "Encerra opção que estava comprada",
    qty: true, price: true, fee: true, tax: false,
  },
  EXERCISED: {
    label: "Exercida",
    hint: "Strike atingido — transfere shares do underlying",
    qty: true, price: false, fee: false, tax: false,
  },
  EXPIRED: {
    label: "Virou pó",
    hint: "Vencimento sem exercício — posição zera, sem cash",
    qty: false, price: false, fee: false, tax: false,
  },
}
```

**3.4 Preview block — adapta**

Para `EXPIRED`: preview mostra "posição zera · sem cash flow ·
prêmio inicial vira receita realizada de OPTION_PREMIUM (Spec 17
sintético)".

Para `EXERCISED`: preview mostra "{qty} shares de {underlying.ticker}
transferidas · cash flow = strike × qty {sign por PUT vs CALL}".

Para `SELL_TO_CLOSE` / `BUY_TO_CLOSE`: igual aos BUY/SELL atuais
mas com label adaptado.

**3.5 Validação**

- `EXPIRED` não permite data futura (vence hoje ou no passado).
- `EXERCISED` não permite data futura nem antes do `option.opened_at`.

### 4. Backend — endpoints novos pra closers

O `POST /options` da Spec 17 só lida com **open** (`SELL_OPEN` /
`BUY_TO_OPEN`). Pra os closers:

- `SELL_TO_CLOSE` / `BUY_TO_CLOSE` → cria `AssetMovement` normal
  via `POST /asset-movements` (já existe). Frontend só precisa
  enviar o `movement_type` correto.
- `EXERCISED` → **novo endpoint** `POST /options/{id}/exercise`
  que cria:
  1. `AssetMovement(type=EXERCISED, qty, fee=0, price=0)` na opção.
  2. `AssetMovement(type=BUY ou SELL, qty=opt.qty × contract_size,
     price=opt.strike_price)` no underlying (BUY se PUT exercida
     contra você; SELL se CALL exercida).
  3. Marca `opt.is_active = False`.
- `EXPIRED` → **novo endpoint** `POST /options/{id}/expire` que
  cria:
  1. `AssetMovement(type=EXPIRED, qty=0, price=0)` na opção.
  2. Marca `opt.is_active = False`.
  3. **Não toca no underlying** nem em cash.

Por que endpoints dedicados pra `EXERCISED` / `EXPIRED`: a
operação é composta (2 movements + flag) e merece atomicidade no
backend, não no frontend.

### 5. Pattern doc — `docs/compound-create-pattern.md`

Documentar o padrão pra próximos compound creates (FIXED_INCOME,
ETF novo):

- Cada classe complexa ganha seu próprio composer dedicado.
- Cada composer vira um item no `NovoButton` (grupo
  Investimentos).
- Backend expõe endpoint compound (`POST /options`, futuro
  `POST /fixed-income`).
- Frontend NÃO tenta composer genérico — campos próprios demais.
- Quando o grupo passar de 5 items, subdividir em
  "Movimentos" e "Novos ativos".

## Out of Scope

- **Composer dedicado de Renda Fixa** (FIXED_INCOME compound).
  Sai numa Spec 37 ou similar quando o user pedir.
- **Edição de opção existente** (mudar strike, vencimento de
  uma opção já criada). Hoje não existe e não vai existir
  nesta spec.
- **Roll de opção** (fechar uma + abrir outra novo strike/vencto
  no mesmo gesto). Bom UX, mas merece spec própria com workflow
  de wizard.
- **Bulk import de opções** (CSV de várias opções no upload —
  cobrir via Spec 38 LLM extraction quando ela ficar pronta).
- **Calculadora de margem** ou "quanto preciso ter na conta pra
  vender essa CSP" — feature de decision support, outro escopo.

## Acceptance Criteria

1. `NovoButton` mostra item "Opção" no grupo Investimentos,
   entre "Lançamento" e "Provento", com shortcut "O".
2. Click em "Opção" abre `OptionModal` standalone com asset
   picker como primeiro campo.
3. Asset picker filtra `klass IN (STOCK, REIT, ETF)` e ordena
   por ticker A-Z.
4. Após selecionar underlying, restante do modal funciona
   idêntico ao fluxo atual (parser de ticker, herança de
   `account_id`, etc).
5. Quando `OptionModal` é invocado de `AssetDetail` (com prop
   `underlying`), asset picker **não** aparece — comportamento
   atual preservado.
6. Botão "Salvar e abrir outra" persiste a opção, limpa
   `ticker`/`strike`/`expiration`/`quantity`/`price`/`fee`/`notes`,
   mantém `underlying` e `optionType`, e foca em `ticker`.
7. Toast de confirmação aparece após cada save.
8. `MovementComposer` com asset OPTION mostra grid 2×2 com
   `SELL_TO_CLOSE / BUY_TO_CLOSE / EXERCISED / EXPIRED`.
9. `MovementComposer` com asset não-OPTION mantém grid 3×2
   atual sem regressão.
10. `EXPIRED` no MovementComposer:
    - Não pede preço nem qty (campos hidden).
    - Submit cria `AssetMovement(type=EXPIRED, qty=0, price=0)`
      e marca asset `is_active=False`.
    - Preview mostra "posição zera, sem cash flow".
11. `EXERCISED` no MovementComposer:
    - Submit dispara `POST /options/{id}/exercise` que cria
      2 movements (opção + underlying) atomicamente.
    - Preview mostra cash flow estimado + shares transferidas.
12. `SELL_TO_CLOSE` / `BUY_TO_CLOSE` criam `AssetMovement`
    normal via endpoint existente.
13. Detecção contextual no `NovoButton`: rota `/ativo/{id}` com
    asset STOCK/REIT/ETF sugere "Opção" como default.
14. `pytest -q` verde (≥6 testes novos cobrindo
    exercise/expire endpoints + validação de datas).
15. `npm test` verde (≥4 testes novos cobrindo OptionModal
    standalone, asset picker, MovementComposer dinâmico).
16. `npm run build` limpo.
17. `docs/compound-create-pattern.md` criado.
