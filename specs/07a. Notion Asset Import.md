# Spec 07a — Notion Asset Import

## Status: 📝 Planned

## Overview

One-shot ETL script that reads the user's existing Notion database **"DB IG Ativos"** (Investidor-Geek workspace) and populates the Numis-Geek `asset` table for the "Família Ambrosio" workspace with real data. Output doubles as a local JSON snapshot so a DB reset doesn't require re-hitting Notion.

This is a **utility, not a product feature**. No UI. Run from CLI by the owner. Idempotent enough to re-run, but not designed for continuous sync — the source of truth shifts to Numis-Geek after the first import.

This is the "07a" half of a three-part sequence: **07a (assets) → 07b (Lançamentos / entries model + UI) → 07c (Notion transactions import)**. Splitting into 07a first lets us validate the Asset model against real data before building anything that depends on it.

---

## Non-goals (explicitly deferred)

- Continuous bidirectional sync with Notion.
- UI for triggering imports.
- Importing Lançamentos, Proventos, Eventos, Apuracoes, Carteira (those are 07c or later specs).
- Importing Notion derived data (formulas, rollups: PM, YoC, DY, current value, etc.).
- Importing the 12 example assets currently in the seed — those are removed by this spec (see "Seed cleanup").

---

## Source schema (Notion)

**Database:** `DB IG Ativos` (`collection://b9a053d1-064b-4ef0-985a-1d3482799462`)

Properties consumed (formulas/rollups ignored):

| Notion property | Type | Used for |
|---|---|---|
| `Ticker` | title (text) | `asset.ticker` (or generated identifier — see "Tickerless rows") |
| `Nome do Ativo` | text | `asset.name` |
| `Tipo do Ativo` | select (`Cotado`/`Não Cotado`) | sanity check; informs `subtype` fallback |
| `Classe` | relation → DB IG Classes | maps to `asset.asset_class` |
| `País` | relation → DB IG Países | informs class disambiguation (BR vs US) and `currency` |
| `IF` | relation → DB IG Instituição Financeira | maps to `asset.financial_institution_id` |
| `Posição Zerada` | formula (checkbox) | drives `is_active = not Posição_Zerada` |

The `URL` of each Notion page is recorded in `asset.notes` (or a future `external_id` field) for traceability.

---

## New AssetClass enum values

Three new values added to `AssetClass` to cover Notion-only classes:

| Enum value | Display (PT) | Notes |
|---|---|---|
| `PRIVATE_PENSION` | Previdência Privada | PGBL/VGBL — not a regular FUND because of taxation/regulation. |
| `FGTS` | FGTS | BR-specific forced-savings; treated as its own class for clarity. |
| `CASH` | Dinheiro | Cash position held outside a checking account (e.g., physical cash, broker float). Distinct from `Account.opening_balance`. |

These are added by an Alembic migration (string-enum values, ALTER TYPE on Postgres; SQLite with `op.batch_alter_table` if needed).

## Class & country mapping

`(Notion Classe, Notion País)` → `Numis AssetClass`:

| Notion Classe | País = Brasil | País = EUA |
|---|---|---|
| Ações | `STOCK_BR` | `STOCK_US` |
| FII / REIT | `FII` | `REIT` |
| Tesouro | `FIXED_INCOME` | `BOND` |
| Renda Fixa / Bonds | `FIXED_INCOME` | `BOND` |
| Fundo / ETF | `ETF` if ticker length ≤ 6 and matches `^[A-Z0-9]+$`, else `FUND` | `ETF` |
| Cripto | `CRYPTO` | `CRYPTO` |
| Carro | `VEHICLE` | `VEHICLE` |
| Imóvel | `REAL_ESTATE` | `REAL_ESTATE` |
| FGTS | `FGTS` | n/a |
| Previdência Privada | `PRIVATE_PENSION` | n/a |
| Dinheiro | `CASH` | `CASH` |

The `Fundo / ETF` heuristic is intentionally rough — easy to refine later via `subtype` if needed.

`Currency` is derived from `País`: BR → BRL, EUA → USD.

## Tickerless rows

Some classes don't have a real ticker in Notion (FGTS, Previdência, Cars, Houses, Cash). The Notion title for those is some manually-chosen string (e.g., "FGTS Itaú", "Apto Pinheiros"). For these:

- `asset.ticker` = `None`
- `asset.name` = the Notion title (or `Nome do Ativo` if richer)
- The original Notion title is preserved in `asset.notes` if it differs from `name`.

For ticker classes (STOCK_*, FII, ETF, REIT, BOND, FUND, CRYPTO), the Notion title **is** the ticker.

---

## `is_active` rule

- `is_active = True` if `Posição Zerada == False` (still holding).
- `is_active = False` if `Posição Zerada == True` (fully sold; preserved for future Lançamento linking).

---

## Specialized rows (`fixed_income_asset`, `physical_asset`)

The Notion source has very little structured data for FIXED_INCOME/BOND/REAL_ESTATE/VEHICLE — none of the indexer/maturity/address fields are stored as Notion properties (they live in unstructured page bodies).

**Decision:** the import creates the parent `asset` row only; the specialized child row (`fixed_income_asset` / `physical_asset`) is **NOT** created in this import. The user fills those in later via the UI.

**Validation impact:** the API normally requires `details` for FIXED_INCOME / REAL_ESTATE / VEHICLE / BOND. The import bypasses that by writing directly via SQLAlchemy (not via the HTTP layer). The DB schema allows the parent `asset` row to exist without the child row (FK is on the child side). The `details`-required rule is API-layer only.

A follow-up task (manual UI work for the user) is to fill in details for FIXED_INCOME / BOND / REAL_ESTATE / VEHICLE / PRIVATE_PENSION / FGTS rows post-import.

## Architecture

### Files

- `scripts/import_notion_assets.py` — main script. CLI entrypoint.
- `data/notion_export.json` — generated snapshot. **Gitignored.** Format below.
- `scripts/seed.py` — modified to (a) drop the 12 example assets, (b) read `data/notion_export.json` if present and seed assets from it.

### Snapshot format (`data/notion_export.json`)

```json
{
  "exported_at": "2026-05-03T20:00:00Z",
  "source_database": "DB IG Ativos",
  "source_workspace": "daniambrosio.notion.so",
  "target_workspace_name": "Família Ambrosio",
  "assets": [
    {
      "notion_id": "18e07f65-cfa4-80f2-...",
      "notion_url": "https://www.notion.so/18e07f65cfa480f2...",
      "ticker": "PETR4",
      "name": "Petrobras PN",
      "asset_class": "STOCK_BR",
      "currency": "BRL",
      "financial_institution_short_name": "XP",
      "is_active": true,
      "subtype": null,
      "notes": null
    },
    ...
  ],
  "skipped": [
    {"notion_id": "...", "reason": "IF 'Nova IF' is a template placeholder"}
  ]
}
```

### Run modes

```
python scripts/import_notion_assets.py --dry-run     # writes JSON, no DB writes; prints summary
python scripts/import_notion_assets.py --apply       # writes JSON AND populates DB
python scripts/import_notion_assets.py --from-json   # skip Notion; just import data/notion_export.json into DB
```

### Idempotency

Match key: `(workspace_id, ticker, financial_institution_id)` for ticker rows; `(workspace_id, name, financial_institution_id, asset_class)` for tickerless rows. On match → update name/is_active/notes; never duplicate.

### Notion access

Uses the Notion MCP server already configured in this environment. The script's CLI invokes the MCP tools via `claude` CLI subprocess **OR** reads from a pre-exported JSON the user feeds in. For simplicity in v1, the user runs the import from inside a Claude Code session ("/import-notion-assets" or just "rode o import") — Claude Code orchestrates the MCP fetches and shells out to the Python script with `--from-json`.

Implementation note: the Python script itself does **not** depend on Notion SDK or MCP. It only reads/writes `data/notion_export.json` and the local DB. The "fetch from Notion" step is a separate utility `scripts/fetch_notion_assets.py` that produces the JSON. This keeps the data-import path testable without Notion access.

## Seed cleanup

`scripts/seed.py` changes:

1. Remove the hard-coded `STARTER_ASSETS` list (the 12 example assets — PETR4, AAPL, SPY, etc.).
2. After the workspace and IFs are seeded, look for `data/notion_export.json`. If present, call the same import logic with `--from-json`. If absent, log a single info message ("No notion_export.json found — skipping asset seed.") and continue.
3. The "Particular" IF added in spec 06 stays.

---

## Validation / safety

- **Dry-run by default** if `--apply` is omitted.
- Print a summary table before exiting: total rows fetched, mapped, skipped (with reasons), would-create vs would-update.
- Exit with non-zero status if any row references an unmapped IF (after fallback to "Particular" warning).
- Never delete assets — only insert and update.
