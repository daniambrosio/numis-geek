# Spec 24 — Price Refresh Background Job

## Status: ✅ Done — 2026-05-24

## Source of truth

- `docs/prototype-deltas-2026-05-23.md` §1.4
- Spec 23 (endpoint `POST /api/prices/refresh`)

## Overview

Roda o refresh de preços **automaticamente todo dia às 18:00
America/Sao_Paulo**, sem precisar de clique do user. Usa APScheduler
embarcado no processo FastAPI; quando migrar pra VPS com gunicorn
multi-worker, revisitar (apenas worker 0 deve agendar — ou trocar por
cron do sistema).

## In Scope

### 1. Dependência

Adicionar ao `pyproject.toml`:

```toml
"apscheduler>=3.10,<4"
```

(APScheduler 4.x está em beta; 3.x é estável e maduro.)

### 2. Scheduler bootstrap

Em `src/numis_geek/scheduler.py` (novo):

```python
from apscheduler.schedulers.background import BackgroundScheduler
from apscheduler.triggers.cron import CronTrigger

def start_scheduler(app):
    sched = BackgroundScheduler(timezone="America/Sao_Paulo")
    sched.add_job(
        run_daily_price_refresh,
        CronTrigger(hour=18, minute=0),
        id="price_refresh_daily",
        replace_existing=True,
        max_instances=1,
        coalesce=True,
    )
    sched.start()
    app.state.scheduler = sched

def stop_scheduler(app):
    if getattr(app.state, "scheduler", None):
        app.state.scheduler.shutdown(wait=False)
```

Wire em `api/app.py` via `lifespan` context manager — start no startup,
stop no shutdown.

### 3. Job runner

```python
def run_daily_price_refresh() -> None:
    """Executes refresh across all workspaces for automated sources.
    Logs audit as `price.refresh.cron` instead of `price.refresh`."""
    with session_scope() as db:
        for ws_id in active_workspace_ids(db):
            summary = refresh_all_automated(db, ws_id, audit_action="price.refresh.cron")
            logger.info(
                "cron price refresh ws=%s ok=%d failed=%d skipped=%d",
                ws_id, summary.ok, summary.failed, summary.skipped,
            )
```

### 4. Audit action variant

`services/price_update.refresh_all_automated()` ganha kwarg
`audit_action: str = "price.refresh"`. Cron passa
`"price.refresh.cron"`; chamadas vindas do endpoint usam o default.

Manual PATCH (spec 28) usa `"price.update.manual"`.

### 5. Disable flag (dev)

`.env` flag `DISABLE_SCHEDULER=true` evita auto-start em dev/testes.
Default = false. Em pytest, sempre disabled (fixture).

### 6. Observabilidade mínima

- Log INFO por workspace processed
- Log WARN se duração > 60s
- Log ERROR + traceback se a iteração inteira falhar (não deve —
  cada asset tem try/except no service)

### 7. Multi-worker note

Adicionar comentário em `scheduler.py` e em
`docs/conceptual-model.md` (ou onde docs de deploy estiverem):

> "Quando rodar com gunicorn multi-worker, este scheduler vai disparar
> N vezes. Soluções: (a) usar `--preload` + worker_id check, (b) migrar
> pra systemd timer / cron do sistema, (c) usar APScheduler com
> jobstore em DB e lock. Decidir antes do deploy VPS."

## Out of Scope

- Múltiplos jobs (só este). Outros virão em specs dedicadas.
- Backfill histórico — spec separada quando precisar.
- Notificações push de sucesso/falha do cron.
- Retry policy sofisticada (se brapi der timeout, falha vai pro
  `errors[]` da execução; próximo dia tenta de novo).

## Acceptance Criteria

1. `apscheduler` instalado e `uv lock` regenerado.
2. App startup loga `"scheduler started: price_refresh_daily next=..."`.
3. Job dispara às 18:00 America/Sao_Paulo (testar manualmente com
   trigger forçado em dev).
4. Audit log tem ≥1 entrada `price.refresh.cron` após execução.
5. `DISABLE_SCHEDULER=true` em pytest impede startup.
6. App shutdown encerra o scheduler limpo (sem hang).
7. `pytest -q` verde.
