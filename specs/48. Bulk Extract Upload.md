# Spec 48 — Bulk Extract Upload (Snapshot-level)

## Status: 📝 Ready for implementation

## Why

O Spec 38 expôs upload de extrato **por pendência individual** — o
usuário precisaria subir o mesmo PDF da XP 20 vezes pra fechar 20
ativos. Não escala. Além disso, o gatilho atual só aparece quando a
heurística `_is_avenue_generic` casa (MANUAL + Avenue + ticker null) —
em produção hoje, zero pendências caem nessa regra.

O backend **já tem** o motor de bulk: `_apply_broker_position` itera
`positions[]` e atualiza N assets por ticker. Mas:
1. Não fecha as pendências correspondentes — só mexe em
   `Asset.current_price`. Resultado: preço atualiza, pendência fica
   aberta, Confirmar fechamento continua bloqueado.
2. Não tem entry point na UI fora do botão por pendência.
3. Não escopa por FI nem distingue casadas/órfãs/não-cobertas.

## Decisões locked

| # | Pergunta | Resposta |
|---|---|---|
| 1 | Escopo do upload | **Por fechamento** (não por FI). Uma área no topo do `<PendencyPanel>`. |
| 2 | "Casadas" | Linhas do extrato cujo ticker bate com um ativo da workspace **que tem pendência aberta** no snapshot atual. |
| 3 | Ativo do extrato que não existe no workspace | **Ignorar com warning** (loga em "Linhas órfãs", não cria asset). |
| 4 | `source_hint` | Esconder do usuário — sempre `BROKER_POSITION` no fluxo bulk. |
| 5 | Anotação de FI | **Opcional no review**: dropdown "Esse extrato é da: …". Vai pro audit log + `Attachment.notes`. |
| 6 | Paste (cmd+V) | **Obrigatório**, igual drag+drop e click. Listener no document enquanto o painel `IN_REVIEW` está montado. |

## In Scope (MVP V1)

### 1. Backend

**1.1 Novo endpoint**

```
POST /snapshots/{snapshot_id}/bulk-extract
  body: { attachment_id: str, institution_short_name?: str }
  → ExtractionJobOut
```

Cria `ExtractionJob` com `snapshot_id` setado, `pendency_id=None`,
`source_hint=BROKER_POSITION`. `institution_short_name` (opcional)
fica em `ExtractionJob.notes` ou nova coluna `institution_short_name`
(decisão na implementação — provavelmente reuso de `notes`).

**1.2 `confirm_extraction` reconhece bulk**

Quando `job.pendency_id is None` AND `job.snapshot_id is not None`,
dispatch pra novo applier `_apply_bulk_to_snapshot` em vez de
`_apply_broker_position`.

**1.3 `_apply_bulk_to_snapshot(db, job, payload)`**

Pra cada `position` no payload:
1. Casa por `ticker_normalized` ou `ticker_raw` em
   `Asset.workspace_id == job.workspace_id`. **Sem fuzzy match em V1**
   (futuro: nome).
2. Se ativo não existe → categoria **orphan** (loga, segue).
3. Se ativo existe mas não tem pendência aberta no snapshot →
   categoria **matched_no_pendency** (atualiza
   `Asset.current_price` ainda assim por bondade? **Decisão: não**.
   Mantém intocado — extrato bulk só fecha pendência. Atualizar preço
   sem pendência é caminho do refresh, não do fechamento).
4. Se ativo + pendência aberta:
   - Atualiza `Asset.current_price`, `price_updated_at`
   - Cria/atualiza `PortfolioSnapshotItem` (quantity de
     `compute_position`, unit_price do extrato, market_values
     recalculados via fx_rate_on(period_end))
   - Marca `SnapshotPendency.resolved_at`, `resolved_by`,
     `resolution_attachment_id=job.attachment_id`,
     `resolution_note="bulk extract"`
   - Emite audit `snapshot.pendency.resolve` com
     `details={"bulk_attachment_id", "institution_short_name", "extraction_job_id"}`
   - Conta em **applied**

Scope por FI quando `institution_short_name` setado: na etapa 3 e 4,
só conta pendência se `asset.account.financial_institution.short_name`
== `institution_short_name`. Pendências de outras FIs são ignoradas
silenciosamente (não viram orphan — só não casam).

**1.4 Response shape**

`ExtractionApplyResult` ganha listas tipadas:

```python
@dataclass
class BulkApplyDetail:
    applied: list[dict]                  # [{pendency_id, asset_id, ticker, new_price, previous_price}, ...]
    matched_no_pendency: list[dict]      # [{asset_id, ticker, extract_price}, ...]
    orphan: list[dict]                   # [{ticker, raw}, ...]
```

Retornado dentro de `applied_count`/`skipped_count`/`errors` ou via
campo novo `details: BulkApplyDetail` no `ExtractionApplyOut` quando
bulk.

### 2. Frontend

**2.1 `<BulkUploadZone />`** — novo componente, renderizado no topo
do `<PendencyPanel>` quando `snap.status === 'IN_REVIEW'`.

Comportamento:
- Dropzone visual (border-dashed amber)
- **Click** abre file picker (accept: image/*, application/pdf, .xlsx, .csv)
- **Drag-and-drop** com highlight no `dragover`
- **Paste (cmd+V)**: useEffect com `document.addEventListener('paste', ...)`,
  pega `event.clipboardData.items`, filtra `kind === 'file'`, converte
  pra `File` (image/png ou whatever)
- Único arquivo por vez (V1). Se user larga 2, processa só o primeiro
  com aviso

Ao receber arquivo:
1. `uploadAttachment('snapshot', snap.id, file)` (escopo snapshot, não asset)
2. `POST /snapshots/{id}/bulk-extract` com `attachment_id`
3. Polling de `GET /extractions/{id}` até `status in (COMPLETED, FAILED)`
4. Abre `<BulkExtractReviewModal />`

**2.2 `<BulkExtractReviewModal />`**

Header: "Revisar extração"

Topo: **Dropdown opcional "Esse extrato é da:"** — popula com FIs
ativas do workspace + "(não informado)" como default. Re-roda
matching client-side quando muda? **Decisão: não em V1.** A escolha
vira só metadata no apply (`institution_short_name` no body do
`confirm`). Re-matching dinâmico fica pra futuro.

3 seções (cards):

1. **Casadas (N)** — verde
   - Checkbox por linha (default: checked)
   - `{ticker} {asset_name} · novo: {fmtBRL(new_price)} · anterior: {fmtBRL(previous_price)} ({delta_pct})`
2. **Pendências não cobertas (M)** — neutro
   - Lista pendências que ainda estarão abertas após aplicar
   - "Vai precisar editar manual ou subir outro arquivo"
3. **Linhas órfãs (K)** — amber
   - "{ticker_raw}: {extract_price} — ativo não encontrado no workspace"

Footer:
- "Cancelar" (libera o job via `rejectExtraction`, fecha modal)
- "Aplicar X resoluções" (chama `confirmExtraction` com
  `{institution_short_name?}` no body)

**2.3 Apply success**

Mostra toast "X pendências resolvidas". Re-fetch pendencies. Modal
fecha. Se sobraram pendências, o painel continua mostrando elas —
user pode subir outro extrato ou editar manual.

## Out of Scope

- **Múltiplos arquivos simultâneos.** V1 = um por vez. Queue futura.
- **Fuzzy matching por nome** (quando ticker é null). V1 = só ticker.
- **Auto-detecção de FI pelo conteúdo.** Dropdown sempre manual.
- **Re-matching dinâmico ao mudar FI no review.** V1 = FI vira só
  metadata.
- **Suporte a `SCREENSHOT_PRICE` ou `OTHER`** no fluxo bulk. Hint
  fixo em `BROKER_POSITION`.
- **Editar valor extraído antes de aplicar.** Se o LLM errou um preço,
  user descarta linha e edita manual. V2 pode adicionar inline edit.

## Acceptance Criteria

1. Upload de um screenshot de extrato cria `Attachment` escopado a
   `snapshot`, dispara `ExtractionJob`, abre modal de review.
2. Cmd+V em qualquer lugar da página (com snapshot IN_REVIEW)
   captura imagem da clipboard e dispara o mesmo fluxo.
3. Apply resolve pendências casadas + atualiza Asset.current_price +
   cria PortfolioSnapshotItem + emite audit por pendência.
4. Anexo do bulk fica linkado em cada pendência via
   `resolution_attachment_id`.
5. Pendências de outras FIs (quando `institution_short_name` é
   passado) não são afetadas.
6. Linhas órfãs (ticker não existe) são reportadas, não criam asset.
7. `pytest -q` verde; `npm run test:run` verde; `npm run build` clean.
