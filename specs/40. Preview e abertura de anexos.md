# Spec 40 — Preview e abertura de anexos

## Status: 📝 Draft — aguarda entrevista em plan mode antes da implementação

## Já entregue fora da spec (2026-05-26)

- **Thumbnail no row** para `kind=image` — `NotesAttachmentsCard`
  agora baixa o blob via fetch autenticado e renderiza um `<img>` 36×36
  no quadrado de ícone (helper `useAttachmentImage`). Atende o caso
  comum (1-10 anexos por registro, ≤1MB cada).

O que ainda fica como Spec 40:
- Endpoint backend que entrega **thumbnail redimensionado** (Pillow) pra
  evitar baixar 10 MB pra renderizar 36×36 quando a listagem cresce.
- Thumbnail de PDF (1ª página via `pypdfium2`).
- Botão "Abrir" + `<AttachmentLightbox>` para visualizar em tamanho
  cheio.
- Endpoint `/inline` (`Content-Disposition: inline`) pra PDF abrir em
  nova aba sem baixar.

## Source of truth

- Feedback do usuário 2026-05-26: depois que Spec 37 entregou upload de
  imagens/PDFs nos detail panels, ficou claro que dois affordances estão
  faltando:
  1. **Preview** — a lista de anexos mostra um ícone genérico (📄 / 🖼️),
     mas o usuário não consegue distinguir "qual screenshot é qual" sem
     clicar pra baixar. Para uma carteira com 3-5 prints por lançamento,
     isso é fricção.
  2. **Abrir** — só existe "Baixar" e "Remover". Pra conferir rápido se
     o anexo é o certo, o usuário quer **abrir o arquivo numa nova aba /
     lightbox**, sem precisar fazer download primeiro.
- `prototypes/index.html:4131-4153` (`AttachmentRow`) — o protótipo só tem
  download + remover; preview e "abrir" são extensões desta spec.

## Overview

Esta spec adiciona dois affordances no `AttachmentRow` (usado em
`NotesAttachmentsCard`, e potencialmente em outras listagens de anexos
futuras):

1. **Thumbnail/preview** no quadrado de ícone:
   - Para `kind=image` (PNG/JPG/WEBP): mostrar miniatura real, não o ícone.
   - Para `kind=pdf`: ficar com o ícone (V1) ou renderizar 1ª página
     como PNG no backend (V2 — depende de `pdf2image`/`pdfium`).
   - Para `kind=csv|other`: ícone (status quo).

2. **Botão "Abrir"** no hover, ao lado de Baixar/Remover:
   - Imagem → abre lightbox modal centralizado com fechamento via Esc /
     clique fora.
   - PDF → abre em **nova aba** com inline disposition (`Content-Disposition:
     inline` em vez de `attachment`).
   - CSV/other → cai pro download (mesma UX de hoje, sem perda).

## In Scope

### 1. Backend

#### 1.1 Endpoint dedicado pra preview/inline

`GET /attachments/{id}/inline` — retorna o arquivo com
`Content-Disposition: inline` (não `attachment`) pra que o navegador
renderize PDF/imagem na aba ao invés de baixar.

Reutiliza toda a lógica de autorização do
`/attachments/{id}/download` (Spec 19).

#### 1.2 Thumbnail por demanda

`GET /attachments/{id}/thumbnail?w=64` — retorna a imagem
redimensionada como PNG/WebP. Lógica:
- `kind=image`: redimensiona via `Pillow` mantendo aspect ratio.
- `kind=pdf` (V2): primeira página → imagem via `pypdfium2` ou
  `pdf2image`. **Se a lib não estiver disponível, retorna 404 — o
  frontend cai pro ícone.**
- Cache no disco (`./data/thumbnails/{id}-{w}.png`) pra evitar
  regenerar a cada lista.

Adicionar `pillow` (sempre) e `pypdfium2` (opcional grupo `pdf`) em
`pyproject.toml`.

### 2. Frontend

#### 2.1 `AttachmentRow` — thumbnail

Quando `kind === 'image'`, substituir o `<KindIcon>` por:

```tsx
<img
  src={`/api/attachments/${att.id}/thumbnail?w=64`}
  loading="lazy"
  className="w-full h-full object-cover"
  onError={() => /* fall back to icon */}
/>
```

Autenticação: o `<img>` não manda Bearer token. Duas opções:
- **A (recomendado)**: emitir cookie HttpOnly/SameSite=Lax na rota do
  thumbnail aceitando o JWT como query-param de uso único — complexo.
- **B (V1 simples)**: o frontend faz `fetch` autenticado, recebe blob,
  cria object URL, usa no `<img src>`. Custa 1 GET por anexo na primeira
  vez (cacheado depois pelo browser via `Cache-Control`).

V1 vai com (B). Migrar pra (A) se aparecer alguma listagem com 50+ anexos.

#### 2.2 Botão "Abrir"

Adicionar entre Download e Remover no `AttachmentRow`:

```tsx
<button onClick={onOpen} title="Abrir">
  <ExternalLink className="w-3.5 h-3.5" />
</button>
```

Comportamento:
- `kind=image` → abre `<LightboxModal>` com a imagem em tamanho original,
  fundo escuro, ESC fecha, atalhos `←/→` se houver vários anexos.
- `kind=pdf` → `window.open('/api/attachments/{id}/inline', '_blank')`.
- `kind=csv|other` → fallback pro download (não quebra).

#### 2.3 `<LightboxModal>` componente novo

Em `frontend/src/components/AttachmentLightbox.tsx`:
- Props: `attachments: AttachmentOut[]`, `startIndex: number`, `onClose`.
- Suporta navegação se receber lista (`←` / `→`).
- Carrega via fetch autenticado (mesmo pattern do download na Spec 37).
- Mostra filename + tamanho no header; download/abrir no footer.

### 3. Cache de thumbnails

Backend grava em `./data/thumbnails/`. Adicionar rota
`DELETE /sysadmin/thumbnails/cache` pra purgar quando precisar (não
expor na UI V1).

## Out of Scope

- **PDF preview de qualquer página além da 1ª** — V1 só renderiza
  página 1.
- **Anotações no PDF** (highlight, comentário) — não é a função do
  app.
- **OCR no preview** — Spec 38 já cobre extração estruturada; isso
  aqui é só visualização.
- **Compartilhamento público via link** — anexos são privados por
  workspace.
- **Edição de imagem inline** (crop, rotate). Se quiser, abre em outro
  app.
- **Galeria com filtros / busca** — fora do escopo desta spec. Lista
  por entidade é suficiente.

## Acceptance Criteria

1. `GET /attachments/{id}/thumbnail?w=64` retorna PNG da imagem
   redimensionada com Cache-Control adequado.
2. Para PDFs sem `pypdfium2` instalado, thumbnail retorna 404 e
   `AttachmentRow` cai pro ícone.
3. `GET /attachments/{id}/inline` retorna o arquivo com
   `Content-Disposition: inline`.
4. `AttachmentRow` mostra miniatura para `kind=image`.
5. Botão "Abrir" aparece no hover; para imagem abre lightbox, para PDF
   abre nova aba, para CSV/outro cai no download.
6. Lightbox abre na imagem clicada e navega com `← / →` se houver
   múltiplas imagens no mesmo registro.
7. `pytest -q` verde (≥3 testes novos: thumbnail, inline, fallback PDF).
8. `npm test` verde (≥2 testes novos: AttachmentRow com preview,
   Lightbox abre/fecha/navega).
9. `npm run build` limpo.

## Pre-implementation interview (Planner agent)

Decisões pra fechar com o user antes de codar:

1. **Cache de thumbnail** — disco do servidor (V1 simples) ou
   regenerar a cada request (zero footprint mas custa CPU)?
2. **Tamanho do thumbnail** — só `w=64` (suficiente pra a row) ou
   suporte a múltiplas (`w=64`, `w=256`)? Múltiplos aumenta cache.
3. **Lightbox keyboard nav** — só `← / →`, ou também `Home/End`,
   `j/k`? Confirmar.
4. **PDF preview no lightbox** — abrir embed (`<embed>` / `<iframe>`)
   dentro do lightbox, ou sempre nova aba? Sugestão: nova aba (V1)
   pra não complicar; lightbox V2 se virar pedido.
5. **Pre-load do próximo na sequência** quando lightbox abre? Melhora
   UX mas custa requests extra.
6. **PDF lib** — `pypdfium2` (rápido, binário nativo) ou `pdf2image`
   (depende do poppler do sistema)? Sugestão: `pypdfium2`.
7. **Fallback de erro** — se o thumbnail backend falha, mostrar
   ícone silencioso, ou um "broken image" indicator?
8. **Permissão** — `/inline` e `/thumbnail` precisam da mesma checagem
   workspace que `/download` (sim). Confirmar autorização: dono do
   workspace + sysadmin.
