## Overview

Automatizar snapshots diários do `numis_geek.db` com rotação. Hoje
backups são manuais (`cp numis_geek.db .bak-YYYY-MM-DD`) e
esquecíveis. Com produção viva e lançamentos diários do user, gap
de 1 dia sem backup = 1 dia de trabalho perdido em caso de
acidente.

## In Scope

### 1. `services/backup.py` — novo

```python
@dataclass
class BackupResult:
    path: Path
    size_bytes: int
    duration_ms: int
    pages_copied: int

def create_backup(
    db_url: str,
    target_dir: Path,
    label: str | None = None,
) -> BackupResult:
    """Atomic SQLite backup via native conn.backup() API.

    Filename: numis_geek-YYYYMMDD-HHMMSS.db
    (or numis_geek-YYYYMMDD-HHMMSS-{label}.db when label provided).
    """

@dataclass
class RotationResult:
    kept: list[Path]
    deleted: list[Path]

def rotate_backups(
    target_dir: Path,
    keep_daily: int = 7,
    keep_monthly: int = 12,
) -> RotationResult:
    """Keep last N daily + last-of-month for N months. Delete rest.

    Parses timestamps from filenames (numis_geek-YYYYMMDD-HHMMSS*.db).
    Files that don't match the pattern are left alone.
    """
```

**Por que SQLite native `backup()` API:**
- Atomic mesmo com writes concorrentes (WAL-safe)
- ~10× mais rápido que dump SQL pra DB pequeno
- File-level binary, restore trivial (`cp .db` de volta)

**Por que não SQL dump (`pg_dump` style):**
- SQLite local não precisa de portabilidade hoje
- Quando migrar pra Postgres no VPS, swap implementação
- Spec não bloqueia esse caminho futuro

### 2. `scheduler.py` — 4º job

```python
BACKUP_JOB_ID = "backup_daily"

def run_daily_backup() -> None:
    """Snapshot do DB + rotação. Idempotente."""
    target = Path("data/backups")
    target.mkdir(parents=True, exist_ok=True)
    result = create_backup(DATABASE_URL, target)
    rotated = rotate_backups(target)
    logger.info(
        "cron daily backup: path=%s size=%dMB pages=%d kept=%d deleted=%d",
        result.path.name, result.size_bytes // (1024*1024),
        result.pages_copied, len(rotated.kept), len(rotated.deleted),
    )
```

Cron: **07:00 SP** (alinhado com início do dia útil, antes que
user comece a usar o app).

### 3. `api/routes/backup.py` — sysadmin force-trigger

```

```

Útil pra ad-hoc (antes de migration, antes de operação manual de
risco, etc).

## Out of Scope

- **Off-host backup** (S3, R2, Drive) — pré-VPS. Por enquanto
  local only.
- **PostgreSQL portability** — quando migrar, swap implementação.
- **Restore CLI** — restore hoje é `cp backup.db numis_geek.db`
  manual. Funciona, não precisa de wrapper.
- **Encryption at rest** — DB é local-only por enquanto.

## Acceptance Criteria

1. `create_backup` cria arquivo `.db` válido em `data/backups/`
   com filename timestampado.
2. Arquivo de backup abre como SQLite válido (`sqlite3 backup.db
   "SELECT COUNT(*) FROM asset"`).
3. `rotate_backups` mantém últimos 7 dias + último de cada mês
   (12 meses), deleta resto.
4. Scheduler tem 4º job `backup_daily` a 07:00 SP.
5. Endpoint `POST /sysadmin/backup` retorna BackupResult; só
   sysadmin pode chamar (403 pros outros).
6. `pytest -q` verde (≥4 tests novos cobrindo backup + rotation +
   scheduler integration).
7. Após primeiro run manual via endpoint, backup aparece em
   `data/backups/` e teste consegue ler.
