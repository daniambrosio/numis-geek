## Why

Bug encontrado por acidente investigando o XP Corporate Light fora do
fechamento de mai/2026 (sessão de 2026-06-06):

```
PETR4 (BRL, STOCK):  qty=1700  avg_native=R$ 25,48  avg_brl=R$ 115,78
XP Corp Light (BRL, PRIVATE_PENSION):  gross=R$ 50.000  total_invested_brl=R$ 268.970
```

`avg_brl` deveria ser igual ao `avg_native` em ativo BRL. `total_invested_brl`
de R$ 50k não pode virar R$ 268k.

### Causa raiz

`compute_position` (positions.py:180,184) faz `* fx` no basis sem checar
moeda do movement:

```python
# Linha 180 (não-cotado, gross-only):
non_cotado_basis_brl += gross * fx     # ← gross JÁ é BRL pra movement BRL
# Linha 184 (cotado, qty × price):
basis_cost_brl += qty * unit * fx       # ← unit JÁ é BRL pra movement BRL
# Linha 219 (distribution income):
total_received_brl += net * fx          # ← net JÁ é BRL pra distribution BRL
```

Por design (memory `multicurrency_fx_rate_design.md`, commit `0da4fb0`),
TODO movement carrega o PTAX do event_date — mesmo BRL — pra permitir
exibir a carteira em USD. O bug é o consumidor (`compute_position`)
multiplicar pelo PTAX sem distinguir moeda.

### Damage radius — DB de prod (2026-06-06)

```sql
-- Movements BRL com fx_rate ≠ 1 (PTAX auto-fillado):
508 / 508 movements BRL ativos
-- Por classe (apenas não-cotado, mais grave porque mexe direto em total_invested):
FIXED_INCOME    5 ativos
PRIVATE_PENSION 2 ativos
FGTS            2 ativos
FUND            1 ativo
-- Cotado BRL (avg_cost_brl inflado): TODAS as STOCK/ETF/REIT BRL com BUY.
```

### Cascata

Onde `average_cost_brl` / `total_invested_brl` aparecem hoje:

- `/portfolio` (api `getPortfolioSummary` → cards de "Total Investido")
- `/dashboard` (mesma fonte)
- `/assets/:id` página de detalhe (Custo Médio, Total Investido)
- `portfolio_snapshot_item.total_invested_brl` (frozen — snapshots
  CLOSED/IN_REVIEW já têm valor errado gravado)
- `/snapshots/:id` página "Posições Congeladas" (lê dos items frozen)

`market_value_brl` dos snapshots é tipado pelo user via pendency → não
afetado. `current_value_brl` na página do ativo usa `fx_rate_on` (PTAX
ao vivo) e respeita a moeda do asset → não afetado.

## Decisões locked

| # | Item | Decisão |
|---|---|---|
| 1 | Fix em positions.py | **Sim** — `compute_position` pula `* fx` quando `movement.currency == BRL`. Aplica em todos os 3 pontos (cotado, não-cotado, distribution). |
| 2 | Limpar `fx_rate` dos movements BRL? | **Não** — design diz que todo movement carrega PTAX (`multicurrency_fx_rate_design`). Só não usar no consumidor BRL. |
| 3 | Recompor snapshot_items.total_invested_brl frozen? | **Sim, opt-in** — script `scripts/56_backfill_snapshot_invested.py` que recomputa só `total_invested_brl` dos items (não toca `market_value_brl`). User roda quando quiser e a Spec 52 fica respeitada porque a coluna afetada é cost-basis, não preço. |
| 4 | Hardening: `portfolio_summary._total_received_brl` e `proventos.py:270` | **Sim** — mesma multiplicação cega. Hoje BRL distributions têm `fx_rate=1.0` (não há damage), mas o código é currency-blind e quebraria se alguém auto-fillar PTAX em distributions BRL no futuro. Trocar pra `_convert_amount` (já existe em proventos.py:325). |
| 5 | Migration de dados | **Nenhuma** — `fx_rate` fica do jeito que está; só o cálculo muda. |

## In Scope

### 1. Backend — `compute_position`

`src/numis_geek/services/positions.py`:

- Linhas 178-187 (basis update): introduzir `mv_fx_effective` que vale
  `fx` quando `r.currency == "USD"` e `Decimal("1")` quando `r.currency
  == "BRL"`. Usar no lugar de `fx` nas 3 multiplicações:
  - `non_cotado_basis_brl += gross * mv_fx_effective`
  - `non_cotado_basis_brl -= gross * mv_fx_effective`
  - `basis_cost_brl += qty * unit * mv_fx_effective`
- Linhas 216-219 (distribution loop): mesmo tratamento usando
  `d.currency` (BRL: fx=1, USD: fx=d.fx_rate).

Nota: linha 233 (`fx = fx_rate_on(...)` pra `current_value_brl` em USD
asset) já está correto e não muda — ela só roda quando `currency ==
"USD"`.

### 2. Backend — `portfolio_summary` e `proventos`

`src/numis_geek/services/portfolio_summary.py`:

- `_total_received_brl` (linha 161) e `_received_by_type` (linha 175):
  hoje fazem `func.sum(Distribution.net_amount * Distribution.fx_rate)`.
  Trocar por `func.sum(case((Distribution.currency=='BRL',
  Distribution.net_amount), else_=Distribution.net_amount *
  Distribution.fx_rate))` ou subir pra agregação Python sobre as rows.

`src/numis_geek/services/proventos.py`:

- Linha 270 (`_sum_distribution_brl`): usar `_convert_amount` (já existe
  no mesmo arquivo, linha 325) ou condicional inline. Hoje rodando
  inofensivo porque BRL dist tem fx=1, mas a defesa custa nada.

### 3. Script de backfill de snapshot items (opt-in)

`scripts/56_backfill_snapshot_invested.py`:

- Itera todos `PortfolioSnapshotItem`.
- Recalcula `total_invested_brl` chamando `compute_position(asset_id,
  as_of=snapshot.period_end_date)` com a versão corrigida.
- Atualiza só `total_invested_brl` (mantém `market_value_brl`,
  `unit_price`, `quantity`, `average_cost_brl` — wait, `average_cost_brl`
  também é frozen-store derivado; mantém ele se igual, atualiza se != ao
  novo).
- Faz `cp numis_geek.db .bak-before-56` antes (padrão
  `migration_snapshot_pattern`).
- Imprime diff por snapshot: quantos items mudaram, antes/depois.
- NÃO toca em snapshots NULL (LIVE PRICE policy do Spec 52 não se
  aplica aqui porque `total_invested_brl` ≠ `market_value`).
