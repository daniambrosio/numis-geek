# Spec 62 — Detecção de variações anômalas no fechamento

## Status: 📝 Design done — pronto pra implementar

## Priority

Alta — se estivesse pronto na Fase 2 do audit, teria pego preventivamente
os bugs do PGBL Flexprev (+R$ 7-8M inflado), Meli Dólar (diagnóstico
errado), BHIA3 (3× stale), Fundo Verde BTG (R$ 81k → R$ 1,72). É a rede
de segurança contra bugs futuros no pipeline de fechamento.

## Why

Hoje o fechamento (spec 14) valida invariantes técnicas (Σ items ==
header, mv_columns_invariant, currency invariante), mas NÃO valida
plausibilidade dos valores por ativo. Um ativo que passa de R$ 80k pra
R$ 1,72 sem lançamento explicativo passa direto pelo CLOSE se todos os
outros checks passarem.

O user quer uma última camada de verificação: comparar cada ativo com
seu valor no snapshot anterior CLOSED e sinalizar variações que fogem
do esperado pra aquela classe. Deltas justificados por movimento no
período (BUY/SELL/BONUS/FULL_REDEMPTION) ou corporate action
(SPLIT/GROUPING/CONVERSION) são auto-suprimidos — só ficam pra revisão
manual as anomalias sem explicação óbvia.

## Regras

### Cálculo do delta MoM

Para cada item do snapshot `S` sendo fechado, compara com o item do
snapshot CLOSED imediatamente anterior (`S_prev`) do mesmo asset:

- **Delta absoluto**: `item.market_value_native - S_prev_item.market_value_native`
- **Delta %**: `abs(delta_abs) / abs(S_prev_item.market_value_native)`
  quando `S_prev.mv_native != 0`; caso contrário `None` (não flagável).

Comparação sempre em **moeda nativa do ativo** (isola PTAX).

### Ativos incluídos vs excluídos

| Caso | Comportamento |
|---|---|
| Ativo NOVO (existe em `S`, não em `S_prev`) | **Skip** — aporte inicial é esperado |
| Ativo ZERADO (existe em `S_prev`, ausente/qty=0 em `S`) | **Skip** — provavelmente saída legítima |
| Ativo com CA no período `[S_prev.period_end, S.period_end]` | **Skip** — SPLIT/GROUPING/CONVERSION justifica |
| Ativo com AssetMovement no período (BUY/SELL/BONUS/SUBSCRIPTION/COME_COTAS/FULL_REDEMPTION) | **Skip** — movimento justifica |
| Restante | Avalia contra threshold |

### Thresholds por asset_class

```python
_SUSPICIOUS_THRESHOLDS = {
    # Value-mode: 15% (patrimônio previdenciário/fundo raramente move mais)
    AssetClass.FUND: Decimal("0.15"),
    AssetClass.PRIVATE_PENSION: Decimal("0.15"),
    AssetClass.FIXED_INCOME: Decimal("0.15"),
    AssetClass.FGTS: Decimal("0.15"),
    AssetClass.REAL_ESTATE: Decimal("0.15"),
    AssetClass.VEHICLE: Decimal("0.15"),
    AssetClass.OTHER: Decimal("0.15"),
    # Cotado: 40% (volatilidade normal, especialmente cripto)
    AssetClass.STOCK: Decimal("0.40"),
    AssetClass.ETF: Decimal("0.40"),
    AssetClass.REIT: Decimal("0.40"),
    AssetClass.CRYPTO: Decimal("0.40"),
    AssetClass.OPTION: Decimal("0.40"),
    # CASH: 20% (saldo raramente pula muito num mês sem transferência)
    AssetClass.CASH: Decimal("0.20"),
}
```

Configurável via env var futuramente (`SUSPICIOUS_DELTA_THRESHOLDS`),
mas hardcoded na primeira versão pra evitar over-engineering.

### SnapshotPendency `SUSPICIOUS_DELTA`

Nova variante em `PendencyReason`:

```python
class PendencyReason(str, enum.Enum):
    ...  # existentes
    SUSPICIOUS_DELTA = "SUSPICIOUS_DELTA"
```

Nova `action_type`:

```python
class PendencyActionType(str, enum.Enum):
    ...  # existentes
    CONFIRM_DELTA = "CONFIRM_DELTA"
```

Campos do `SnapshotPendency` populados:

```python
SnapshotPendency(
    reason=SUSPICIOUS_DELTA,
    action_type=CONFIRM_DELTA,
    detail={
        "previous_snapshot_id": S_prev.id,
        "previous_period_end": S_prev.period_end_date.isoformat(),
        "previous_mv_native": str(S_prev_item.mv_native),
        "current_mv_native": str(item.mv_native),
        "delta_native": str(delta_abs),
        "delta_pct": str(delta_pct),
        "threshold_pct": str(threshold),
        "asset_class": asset.asset_class.value,
    },
)
```

### Detecção — quando roda

`detect_suspicious_deltas(db, snapshot_id)` é chamada:

1. **No `create_snapshot`**, após todos os items serem populados (junto
   com `detect_pendencies` existente).
2. **No `apply_recompute_to_snapshot`**, após o recompute do item —
   se o novo valor cria um delta suspeito, cria pendency.
3. **No `sync_snapshot_items`**, para novos items adicionados.
4. **No endpoint** `POST /snapshots/{id}/recheck-deltas` — recalcula
   pra snapshot IN_REVIEW. Útil quando user edita items manualmente.

### Bloqueio

`confirm_snapshot` já reject se há pendencies open. `SUSPICIOUS_DELTA`
segue a mesma regra — não fecha até `resolved_at` estar setado.

### Resolução

Endpoint novo: `POST /snapshots/{id}/pendencies/{pen_id}/confirm-delta`
com body `{ "note": "optional user note" }`. Marca `resolved_at=now`,
`resolved_by=user`, `resolution_note=note`. Sem mudança nos items —
o user só confirma "sim, esse delta é real".

Além disso, se durante a resolução o user EDITAR o preço via
`update_snapshot_item_price`, a pendency deve auto-fechar se o novo
delta ficar abaixo do threshold. Adicionar hook no `update_snapshot_item_price`
pra re-avaliar SUSPICIOUS_DELTA daquele item.

## KPIs / definição de sucesso

- ≥ 1 SUSPICIOUS_DELTA aberto em pelo menos 1 dos próximos 3 fechamentos
  (validação de que o pipeline achou algo real).
- Zero false positives por movement legítimo (validação do auto-suprime).
- Zero regressão em `confirm_snapshot` de snapshots sem SUSPICIOUS_DELTA.
