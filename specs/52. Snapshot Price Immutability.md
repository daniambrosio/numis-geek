# Spec 52 — Snapshot Price Immutability

## Status: ✅ Done (2026-06-04)

50 testes do snapshot lifecycle passando, incluindo 7 novos cobrindo:
preserve frozen unit_price em recompute, pendency
HISTORICAL_PRICE_REQUIRED em add/sync/recompute pra snapshot antigo,
retry_api bloqueado em snapshot antigo, preview da modal usa preço
frozen. TypeScript limpo, PendencyPanel mostra a nova reason com
mesma UX de MANUAL_SOURCE.

## Why

Specs 14/35 estabelecem que `PortfolioSnapshot` é **frozen** —
period_end_date, fx_rate e items são "verdade histórica" pra
performance MoM/YoY. Spec 51 cravou a regra inversa do "uma vez frozen,
sempre frozen": permite reconciliar snapshots quando user descobre
evento retroativo (compra esquecida, split atrasado).

**Mas a Spec 51 (e vários paths adjacentes) sobrescrevem `unit_price`
do item com `asset.current_price`** — o preço LIVE de hoje. A causa:
`compute_position(as_of=period_end)` em `services/positions.py:222`
retorna `asset.current_price` independentemente do `as_of`. O `as_of`
filtra só movimentos/distribuições, não o preço.

Consequência: cada mexida retroativa (Spec 51) — ou cada add manual
em snapshot antigo — substitui o preço de fechamento do mês pelo preço
de HOJE.

Pior: `PortfolioSnapshotItem.unit_price` é a fonte canônica de preço
histórico no sistema. O endpoint `/assets/{id}/price-history` (Spec 46)
lê daí pra montar o sparkline. Sparkline mostra a série
`(period_end, unit_price)` de cada snapshot. Corromper o `unit_price`
do snapshot 2026-04 com o preço de 2026-06 distorce o sparkline.

Audit log confirma: `snapshot.item.recompute` tem 0 entradas em
produção — bug ainda não disparou, mas qualquer mov retroativa daqui
pra frente dispara.

| Path | Onde |

## Decisões locked (entrevista 2026-06-04)

| # | Pergunta | Resposta |
|---|---|---|
| 1 | Recompute de item existente deve preservar `existing.unit_price`? | **Sim** — frozen. Só qty, average_cost, total_invested e market_value (= new_qty × frozen unit_price) recalculam. |
| 2 | Item novo em snapshot — quando usar `current_price` vs pendency? | **`period_end_date == hoje` → `current_price` (primeira captura). Caso contrário → `unit_price=None` + pendency `HISTORICAL_PRICE_REQUIRED`**. |
| 3 | Retry API em snapshot antigo | **Bloquear** — API retorna LIVE, não period_end. `ValueError` direcionando pra preencher manualmente. |
| 4 | Reason nova ou reusa `MANUAL_SOURCE`? | **Nova: `HISTORICAL_PRICE_REQUIRED`**. Semântica clara, mensagem direcionada no UI. |
| 5 | Aplicar mesma regra em todos os paths? | **Sim** — sync, add, recompute, retry. Regra única. |
| 6 | Phase B (historical fetch via providers) entra nesta spec? | **Não.** Spec 53 separada — esta entrega o esqueleto (preserve frozen + pendency). |

### 3. Service — helper único

`src/numis_geek/services/snapshot.py`:

```python
def _new_item_price_or_pendency(
    snap: PortfolioSnapshot,
    asset: Asset,
    pos: Position,
    *,
    now: datetime,
) -> tuple[
    Decimal | None,  # unit_price
    Decimal | None,  # market_value_native
    Decimal | None,  # market_value_brl
    Decimal | None,  # market_value_usd
    SnapshotPendency | None,  # pendency a persistir (ou None)
]:
    """Decide unit_price + market_values pra um item NOVO.

    Regra:
    - period_end_date == today → usa pos["current_price"] (primeira
      captura, OK).
    - period_end_date != today → unit_price=None + pendency
      HISTORICAL_PRICE_REQUIRED + EDIT_PRICE.

    Caller persiste o item + (opcionalmente) a pendency.
    """
```

### 4. Fix `apply_recompute_to_snapshot` (paths 1188 + 1204)

- Branch **item existente** (`existing is not None`):
  - **NÃO sobrescreve** `existing.unit_price`.
  - Recompute: `quantity`, `average_cost_brl`, `total_invested_brl`.
  - `market_value_native = new_qty × existing.unit_price` quando
    `existing.unit_price` está set; caso contrário deixa None.
  - `market_value_brl/usd` deriva via `snap.fx_rate_usd_brl` (já
    frozen).

- Branch **item novo** (`existing is None`):
  - Usa `_new_item_price_or_pendency`.

### 5. Fix `list_affected_snapshots` (path 1071)

Calcula preview do market_value usando o preço que SERIA persistido:
- Item existente → `existing.unit_price` (frozen).
- Item novo + period_end == hoje → `pos["current_price"]`.
- Item novo + period_end != hoje → None (preview mostra "—").

### 6. Fix `add_snapshot_item` (path 815) e `sync_snapshot_items` (path 907)

Substituem `unit_price=pos["current_price"]` direto pelo helper.

### 7. Fix `retry_pendency_api` (path 1426)
