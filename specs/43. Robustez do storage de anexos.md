# Spec 43 — Robustez do storage de anexos

## Status: 🟡 Parcial — hard delete entregue 2026-05-26; resto pendente

## Source of truth

- Auditoria feita com o usuário 2026-05-26 sobre o storage atual de
  `Attachment` (Spec 19). Três preocupações levantadas:
  1. DELETE só fazia soft delete; arquivo continuava ocupando disco.
  2. `ROOT` do filesystem é hardcoded (`./data/attachments`), sem env
     var nem UI.
  3. Defesa em profundidade do isolamento por workspace — `absolute_path`
     valida só path-traversal, não que o prefixo da subpasta bate com
     o `att.workspace_id`.

## Já entregue (2026-05-26)

### Hard delete imediato

`DELETE /attachments/{id}` agora:

- Apaga o arquivo do filesystem (`attachment_storage.delete`).
- Apaga a linha do banco (`db.delete(att)`).
- Recusa com **409 Conflict** se algum `ExtractionJob` referencia o
  attachment (FK constraint `nullable=False` protege contra cascata
  silenciosa).
- Registra audit log com `action="attachment.deleted"` + snapshot dos
  metadados (filename, storage_key, size, workspace) — assim mesmo
  sem a linha, há rastro auditável.
- Tratamento gracious: se o arquivo já sumiu do disco
  (`FileNotFoundError`), a linha ainda é dropada (FS já estava fora
  de sync).

Testes em `tests/test_attachments.py`:
- `test_delete_hard_removes_row_and_file` — confere que tanto a linha
  quanto o `.png` do disco somem.
- `test_delete_missing_returns_404` — segunda chamada DELETE no mesmo
  ID retorna 404.

Arquivos modificados:

## Ainda pendente — escopo desta spec

### 1. `ROOT` configurável via env — *aguarda discussão de tradeoffs*

Status: **decisão pendente, ver §"Tradeoffs em aberto".** Esboço técnico
abaixo pra referência caso a decisão seja "implementa simples".

`src/numis_geek/services/attachment_storage.py` linha 16:

```python
ROOT = Path("./data/attachments")  # hardcoded
```

Trocaria por (env opcional — comportamento preservado quando ausente):

```python
import os

_DEFAULT_ROOT = Path("./data/attachments")

def _resolve_root() -> Path:
    """ATTACHMENTS_ROOT, when set and non-empty, overrides the default.
    Missing/empty env keeps the legacy hardcoded path so existing
    installs are unaffected."""
    raw = os.environ.get("ATTACHMENTS_ROOT", "").strip()
    return Path(raw) if raw else _DEFAULT_ROOT

ROOT = _resolve_root()
```

**Regras**:
- App **NÃO** move arquivos sozinho quando `ATTACHMENTS_ROOT` muda.
  Isso é responsabilidade do sysadmin (parar app → `rsync -av
  old/ new/` → atualizar env → reiniciar). Migração automática teria
  risco de corromper estado parcialmente migrado, perder permissões,
  ou explodir em paths inacessíveis.
- Documentar no `.env.example` que a env é opcional e que mudá-la
  exige migração manual.
- Não expor na UI sysadmin — só via env do processo, lido no boot.

Pra deploy futuro em VPS / S3, esse mesmo ponto vira o switch pra
trocar o backend de storage (FS local vs object storage), per o
comentário existente no `attachment_storage.py:5`.

## Tradeoffs em aberto — discutir antes de implementar §1

Hardcoded é ruim, mas as alternativas têm seus problemas:

| Aproximação | Vantagem | Desvantagem |
|---|---|---|
| **Hardcoded** (estado atual) | Zero superfície de erro; o app só funciona com paths que ele mesmo criou. | Inflexível pro deploy; força o usuário a rodar do mesmo cwd. |
| **Env var opcional + sem auto-migração** | Permite VPS/Docker/cwd diferente. Sysadmin controla migração. | Sysadmin precisa saber: (a) que precisa mover arquivos; (b) que existem `storage_key` relativos no DB. Esquecer = anexos órfãos. |
| **Env var + detecção de "ROOT mudou desde o último boot"** | App avisa logo no boot que arquivos esperados não existem. | Estado a manter (último ROOT visto). Complica o boot. |
| **Sysadmin UI configurável** | Bonito, mas... | Mudança em runtime quebra `storage_key` em uso. Salvar-só-no-DB também quebra reinício. Não vale a pena. |
| **Sempre absoluto, sem default** | Força o operador a pensar no path. | Mata "git clone + ./dev.sh" → friction pra contribuidor novo. |

**Pontos pra alinhar**:

1. Detectar quando o path mudou: o app **deve** falhar em boot
   ruidosamente ("ROOT mudou e há `storage_key`s no DB que não
   existem em disco") ou continuar silencioso (que vira download
   404 caso a caso)? Cabeça do sysadmin vs UX do usuário comum.
2. Migração de arquivos é fora do escopo do app (consenso). Mas o
   app pode fornecer um **CLI helper** que detecta o mismatch
   (`python -m scripts.check_attachments`)? Ou puramente
   responsabilidade externa (rsync + checksum manual)?
3. O `storage_key` é relativo (`{ws}/{uuid}.png`). Se um dia
   virarmos object storage, vira chave de bucket. Mantém formato
   ou vira URL completa? — provavelmente é outra spec.
4. Concurrency: se 2 processos do app rodam (improvável hoje, mas
   pode ser dev + prod no mesmo host), com ROOTs diferentes, cada
   um trava seu próprio `data/`? Os dois precisam apontar pro
   mesmo path? — corner case, mas vira pegadinha em Docker
   Compose.

**Recomendação minha** (não decisão): env var opcional + sem
auto-migração + CLI `scripts.check_attachments` que reporta
mismatch. Mantém simples, dá ao sysadmin a ferramenta de
verificação sem o app tomar decisão sozinho.

### 2. Validação extra: storage_key deve estar dentro do workspace

Hoje `absolute_path()` aceita qualquer `storage_key` que não escape
do `ROOT`. Adicionar uma checagem complementar quando a chamada
acontece no contexto de uma row específica:

```python
def absolute_path_for(att: Attachment) -> Path:
    """Like absolute_path() but also asserts the key sits under the
    attachment's own workspace subdir."""
    expected_prefix = f"{att.workspace_id}/"
    if not att.storage_key.startswith(expected_prefix):
        raise ValueError(
            f"Attachment {att.id} storage_key {att.storage_key!r} "
            f"escapes its workspace subdir",
        )
    return absolute_path(att.storage_key)
```

Substituir as 2 chamadas em `routes/attachments.py` (`download_attachment`
e `delete_attachment`) por essa variante. Pra import de fora (ex:
extraction service) idem.

É **defesa em profundidade** — a checagem de autorização nas rotas
já bloqueia cross-workspace via API. Esta verificação só pega o
caso patológico de uma row corrompida apontando pra outro workspace.

### 3. Detecção de arquivos órfãos no FS (V2 opcional)

Job de housekeeping (APScheduler ou comando CLI) que lê o conteúdo
de `ROOT/{workspace_id}/`, compara com a tabela `attachment`, e
reporta arquivos que existem no FS mas não têm `storage_key`
correspondente. Apenas reporta — não apaga (segurança).

Acionado manualmente em produção via `python -m scripts.audit_attachments`.
Útil pra detectar bugs (race entre delete da row + delete do FS).

## Out of Scope

- **Migrar pra object storage (S3/R2)** — out-of-scope. Quando vier
  o deploy em VPS, vira spec própria com migration de arquivos.
- **Compactação automática de imagens** antes de salvar — escopo da
  Spec 40 (thumbnails) cobre o caso de "imagem grande pra preview
  pequeno"; pra reduzir storage real, vira spec própria.
- **Cota de storage por workspace** — sem feature de billing/limit hoje.
- **Encriptação at-rest** dos anexos — entra na spec geral de
  segurança pré-VPS (memory `pending_security_spec`).

## Acceptance Criteria (pendentes)
