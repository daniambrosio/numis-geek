## Overview

Spec 14 shipped `PortfolioSnapshot` as a simple photograph (CLI + manual
endpoint). This spec extends the same model with a **monthly review
workflow**:

1. **Lifecycle** on the existing snapshot: `SCHEDULED` → `IN_REVIEW` →
   `CLOSED` (with reopen path).
2. **Per-asset pendency tracking**: which data source is missing, why,
   and how to resolve it (retry API / edit manual / upload extract).
3. **Auto-snapshot job** that runs on the first business day of each
   month, captures prices for the last business day of the previous
   month, and creates the snapshot in `IN_REVIEW` when manual sources
   are pending (or `CLOSED` if every automated source returned OK).
4. **Dedicated UI** — `/snapshots` (list + status of the next scheduled
   job) and `/snapshots/{ym}` (frozen report + pendency panel), plus a
   discreet CTA banner on Dashboard.

The existing CLI (`scripts/create_snapshot.py`) stays for retroactive
operations.

## In Scope

### 1. Schema — extend the existing snapshot model

> One concept end-to-end: `PortfolioSnapshot` already exists and powers
> `/portfolio` and `/dashboard`. We add lifecycle + pendency fields
> to the same model, not a parallel "closing" entity.

**1.1 `SnapshotStatus` enum**

```python
# src/numis_geek/models/portfolio_snapshot.py
class SnapshotStatus(str, enum.Enum):
    SCHEDULED  = "SCHEDULED"   # job pending (hasn't run yet)
    IN_REVIEW  = "IN_REVIEW"   # job ran, but manual/upload data still missing
    CLOSED     = "CLOSED"      # frozen, ready for downstream consumers
```

**1.2 New columns on `PortfolioSnapshot`**

```python
status: Mapped[SnapshotStatus] = mapped_column(
    Enum(SnapshotStatus), nullable=False, default=SnapshotStatus.CLOSED,
)
closed_at:    Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
closed_by:    Mapped[str | None]      = mapped_column(String(36), nullable=True)

auto_run_at:  Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
```

- `status` default = `CLOSED` so existing 28 Notion-imported rows stay
  consistent (production_day_0 — never lose this data).
- `closed_at` / `closed_by` = when someone hit "Confirm" (or the job
  auto-closed because no pendencies were detected).
- `scheduled_at` = when the job was scheduled to run.
- `auto_run_at` = when the job actually ran (delta = lag).

**1.3 New table `SnapshotPendency`**

```python
class PendencyReason(str, enum.Enum):
    API_FAILED      = "API_FAILED"      # automated source failed; retry possible
    MANUAL_SOURCE   = "MANUAL_SOURCE"   # MANUAL asset; user must edit price
    UPLOAD_REQUIRED = "UPLOAD_REQUIRED" # MANUAL asset whose price comes from an extract
    STALE_PRICE     = "STALE_PRICE"     # price_tier=OLD at period_end (see §2.2 rules)

class PendencyAction(str, enum.Enum):
    RETRY_API    = "RETRY_API"
    EDIT_PRICE   = "EDIT_PRICE"
    UPLOAD_FILE  = "UPLOAD_FILE"

class SnapshotPendency(Base):
    __tablename__ = "snapshot_pendency"
    id: Mapped[str] = mapped_column(String(36), primary_key=True, ...)
    snapshot_id: Mapped[str] = mapped_column(ForeignKey("portfolio_snapshot.id"), nullable=False)
    asset_id:    Mapped[str] = mapped_column(ForeignKey("asset.id"), nullable=False)
    reason:      Mapped[PendencyReason]  = mapped_column(Enum(PendencyReason), nullable=False)
    action_type: Mapped[PendencyAction]  = mapped_column(Enum(PendencyAction), nullable=False)
    detail:      Mapped[str | None]      = mapped_column(Text, nullable=True)
    resolved_at: Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
    resolved_by: Mapped[str | None]      = mapped_column(String(36), nullable=True)
    resolution_note: Mapped[str | None]  = mapped_column(Text, nullable=True)
    created_at:  Mapped[datetime] = mapped_column(DateTime, ...)

    __table_args__ = (
        Index("ix_pendency_snapshot", "snapshot_id"),
        UniqueConstraint("snapshot_id", "asset_id", name="ux_pendency_snap_asset"),
    )
```

Rule: **one asset has at most one pendency per snapshot**. If a retry
fails and the action shifts to manual edit, the same row is updated.

**1.4 Migration**

- Adds the new columns on `portfolio_snapshot` (default `CLOSED` for

- Creates `snapshot_pendency` table.
- Creates enums via `sa.Enum(...)`.

### 2. Service — lifecycle + pendency detection

**2.1 `services/snapshot.py` — extend `create_snapshot`**

```python
def create_snapshot(
    db: Session, *, workspace_id: str, period_end: date,
    user_id: str | None = None,
    source: SnapshotSource = SnapshotSource.MANUAL,
    initial_status: SnapshotStatus = SnapshotStatus.CLOSED,
    replace_if_exists: bool = True,
) -> SnapshotResult:
    """
    - Compute positions (compute_position with as_of=period_end).
    - Create PortfolioSnapshot with initial_status.
    - For each active asset with qty > 0: detect pendencies and create
      SnapshotPendency rows.
    - If initial_status=CLOSED but pendencies > 0:
        downgrade to IN_REVIEW.
    - Return SnapshotResult with items_count + pendencies_count.
    """
```

**Behavior change** vs. today: `create_snapshot` currently silently
overwrites with `replace_if_exists=True`. New rule:
- `replace_if_exists=True` + existing snapshot in `CLOSED` →
  requires `force_reopen=True` (guard against accidental overwrite).
- Otherwise behaves like today.

**2.2 `detect_pendencies(db, asset, period_end) → list[SnapshotPendency]`**

Rules based on `PriceSource` (Spec 22, current enum
`BRAPI / FINNHUB / COINBASE / TESOURO / MANUAL`):

| `asset.price_source`           | Behavior                                                                  |
|---|---|
| `BRAPI`, `FINNHUB`, `COINBASE`, `TESOURO` | Try refresh; if it failed → `API_FAILED`. If `price_tier == OLD` at period_end → `STALE_PRICE`. |
| `MANUAL`                       | Always pendency. `UPLOAD_REQUIRED` when heuristic matches (asset notes/`external_source` hint at an Avenue-generic or broker-extract origin); otherwise plain `MANUAL_SOURCE`. |
| `NULL` / no source             | Skip — asset has no expected price (CASH, FGTS legacy). |

`asset.quantity_held == 0` at `period_end` → not in the snapshot at all
(no pendency).

> **STALE_PRICE threshold:** only `price_tier == OLD` triggers a
> pendency. `FRESH` and `STALE` tiers are tolerated (per spec 22 they
> mean "older than 24h but recent enough"). The rule is intentionally
> permissive — strict re-pricing of FRESH assets would create noise.

> **UPLOAD heuristic:** v1 picks UPLOAD when
> `asset.external_source == 'NOTION'` and the asset's institution short
> name is "Avenue" and `ticker IS NULL` (the generic Avenue
> `rendimentos` case). Everything else falls through to plain
> `MANUAL_SOURCE`. Heuristic is meant to be refined as we learn — the
> spec doesn't lock it in.

**2.3 Confirm / Reopen**

```python
def confirm_snapshot(db, *, snapshot_id, user_id) -> SnapshotResult:
    """
    - Verify no pendency with resolved_at IS NULL.
    - Recompute totals (prices may have moved after creation).
    - Status → CLOSED, closed_at=now, closed_by=user_id.
    - Emit audit (Spec 03): action='snapshot.confirm', resource='snapshot/{id}'.
    - Raises ValueError when pendencies still open.
    """

def reopen_snapshot(db, *, snapshot_id, user_id, reason: str) -> None:
    """
    - Status → IN_REVIEW (works on any snapshot regardless of age).
    - Re-detect pendencies vs. current price state.
    - Emit audit (Spec 03): action='snapshot.reopen', details={reason}.
    """
```

No restriction on which snapshot can be reopened. Reopening an older
snapshot does NOT cascade-recompute the snapshots after it — the user
accepts the historical drift consciously. UI shows a warning when
reopening a snapshot older than the most recent CLOSED one.

**2.4 Pendency resolution**

```python
def resolve_pendency(
    db, *, pendency_id, user_id,
    new_price: Decimal | None = None,
    file_id: str | None = None,
    note: str | None = None,
) -> SnapshotPendency:
    """
    - Mark resolved_at, resolved_by.
    - If new_price: update Asset.current_price + PortfolioSnapshotItem
      for this asset on this snapshot.
    - If file_id: attach via Attachment (spec 19). LLM extraction is
      out of scope (deferred — see "Out of Scope").
    - Recalculate PortfolioSnapshot totals.
    - Emit audit: action='snapshot.pendency.resolve', resource='pendency/{id}'.
    """
```

**4.2 Job runner**

```python
# src/numis_geek/jobs/snapshot_auto.py
def run_monthly_snapshot(db: Session, *, target_ym: str | None = None) -> list[SnapshotResult]:
    """
    - target_ym=None → previous month relative to today.
    - For each active workspace:
        1. period_end = last_business_day_of_month(target_ym)
        2. Skip if a CLOSED snapshot already exists (idempotent).
        3. Trigger bulk refresh via existing refresh service (spec 23).
        4. Call create_snapshot(..., source=AUTOMATED,
           initial_status=CLOSED) — downgrades to IN_REVIEW
           automatically if pendencies are detected.
        5. Audit log entry.
    - Returns SnapshotResult per workspace.
    """
```

**4.3 Scheduler**

Reuses the in-process APScheduler from spec 24 (`src/numis_geek/scheduler.py`).
Adds a second job:

```python
# in scheduler.start_scheduler()
sched.add_job(
    run_monthly_snapshot,
    CronTrigger(day=1, hour=6, minute=30),
    id="monthly_snapshot",
    replace_existing=True, max_instances=1, coalesce=True,
)
```

Runs `06:30 America/Sao_Paulo` on the 1st of every month. The job is
**idempotent**: re-running it on the same day skips workspaces whose
target_ym already has a CLOSED snapshot. If day=1 happens to be a
weekend/holiday, the job still fires — `last_business_day_of_month`
returns the correct business day for `period_end`.

**No auto-close timeout.** A snapshot left in `IN_REVIEW` stays there
forever until the user clicks Confirm.

**4.4 CLI keeps working**

```bash
python -m scripts.create_snapshot --workspace "X" --period 2026-04-30
# new flag — same flow as the job (detects pendencies):
python -m scripts.create_snapshot --workspace "X" --period 2026-04-30 --auto
```

## Acceptance Criteria

1. Migration applies cleanly on the production DB (28 existing Notion
   snapshots all end up as `status=CLOSED`, `closed_at=created_at`).
2. `create_snapshot` detects pendencies in 3 categories
   (API_FAILED, MANUAL_SOURCE, STALE_PRICE; UPLOAD_REQUIRED via
   heuristic). `FRESH` and `STALE` API tiers do NOT generate pendencies.
3. `POST /snapshots/{id}/confirm` returns 409 when a pendency is open,
   200 when all resolved.
4. `POST /snapshots/{id}/reopen` works on ANY snapshot regardless of age
   and re-detects pendencies. Audit logged.
5. `run_monthly_snapshot` is idempotent — running it twice on the
   same day doesn't duplicate snapshots.
6. Scheduler registers the job at startup (`scheduler started:
   monthly_snapshot next=...`).
7. CLI `scripts/create_snapshot ... --auto` creates `IN_REVIEW` when
   manual assets are present.
8. `/snapshots` renders the history + in-review hero + next-job card.
9. `/snapshots/{ym}` in `IN_REVIEW` shows the PendencyPanel with
   functional Retry/Upload/Edit buttons; Confirm disabled while
   pendencies open.
10. Dashboard shows CTA banner when an `IN_REVIEW` snapshot exists.
11. Sidebar shows amber `!` when an `IN_REVIEW` snapshot exists.
12. Confirm / reopen / resolve / retry-api each emit a
    `snapshot.*` audit row.
13. `pytest -q` green (≥14 new tests).
14. `npm run test:run` green (≥4 new tests).
15. `npm run build` clean.
