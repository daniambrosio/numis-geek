# Spec 61a — Decision Support: Target Allocation

## Status: ✅ Done (2026-06-26)

## Priority

Foundational — desbloqueia 61c (Markowitz precisa de targets como input).

## Why

Pra que o Suporte à Decisão consiga:
- Mostrar gap "alocação atual vs alvo" por classe e por país.
- Usar metas por classe como equality constraints do Markowitz.
- Usar caps por país como inequality constraints do Markowitz.

A entidade `target_allocation` é workspace-scoped e editável pelo admin
via página de settings dedicada (`/admin/target-allocation`).

## Scope

### Backend

- Tabela nova `target_allocation` com `(workspace_id, dimension, key,
  target_pct)` única por trio (UNIQUE constraint).
- Enum `TargetAllocationDimension`: `CLASS` | `COUNTRY`.
- Service `services/target_allocation.py` com `get_targets`,
  `upsert_targets` (transação delete-then-insert), `validate_entries`.
- Validação: soma ∈ [1.0 ± 0.0001] por dimensão; sem keys duplicados;
  CLASS keys validados contra `AssetClass` enum; COUNTRY keys
  validados como ISO-2 alfa (uppercase normalizado).
- Audit log `target_allocation.update` em todo upsert.
- Endpoints REST:
  - `GET /api/workspaces/{id}/target-allocation` — qualquer member do
    workspace; sysadmin pode passar workspace_id arbitrário.
  - `PUT /api/workspaces/{id}/target-allocation` — admin/sysadmin only.

### Frontend

- Página nova `pages/admin/TargetAllocation.tsx` em rota
  `/admin/target-allocation`.
- Item de menu novo na seção Admin (`Target` icon, lucide).
- Layout: PageHeader + tab `Por classe`/`Por país` + tabela editável.
- Tabela: select de key (filtrando opções já usadas) + input `%`
  (number step 0.01) + botão remover.
- Footer com badge de soma — verde se 100%, vermelho caso contrário.
- Botões: Adicionar / Distribuir igualmente / Restaurar padrão (preset
  hardcoded de exemplo) / Salvar metas.
- Modal de confirmação ao salvar; ESC fecha (`useEscapeKey`).
- Toast de sucesso.
- API client em `lib/api.ts`: `api.getTargetAllocation`,
  `api.putTargetAllocation` + tipos.

### Tests

- Backend pytest (`test_target_allocation_service.py` +
  `test_target_allocation_route.py` — 30 testes): validação (CLASS keys
  inválidas, COUNTRY ISO-2 inválido, soma ≠ 1, negativo, > 1, duplicado),
  upsert (insert, replace, isolamento entre dimensões, normalização de
  case), get (workspace vazio, ordenação), RBAC (member do workspace,
  cross-workspace 403, sysadmin), audit log criado, persistência em GET
  subsequente.
- Frontend vitest (`pages/admin/TargetAllocation.test.tsx` — 7 testes):
  empty state, render com dados, edição updates soma + badge, save flow
  com modal de confirmação, ESC fecha modal, troca de tab, distribuir
  igualmente.

## Decisions locked (2026-06-26)

- **Dimensões v1: CLASS + COUNTRY apenas.** Currency target fica v1.5+
  (não há demanda imediata; portfolio summary já cobre split por moeda).
- **Asset-level targets fora do escopo.** Suno-style "preço-teto +
  target weight por ativo" só faz sentido junto com Valuation (61b) —
  pode entrar em 61b ou 61.5.
- **Sem seed automático do Notion v1.** User cadastra manualmente na
  página. Importação posterior se fizer sentido.
- **Reserva de emergência fora.** É escopo do v1.5 (rationale §10 #3
  fica aguardando).

## Out of Scope (defer)

- Seed/import dos targets do Notion.
- Targets por ativo individual.
- Currency dimension (BRL/USD).
- Targets por sector/segmento.
