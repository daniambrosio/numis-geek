# Spec 55 — Observability Hardening for Production

## Status: ✅ Done (2026-06-04)

8 testes verdes. RequestIDMiddleware atribui UUID curto e expõe via
header X-Request-ID. AuditMiddleware agora loga TODO 4xx/5xx (inclui
GET) com action `http.<status_class>.<método>.<path>` e details
incluindo request_id + status + path. RotatingFileHandler escreve em
data/logs/numis.log (10MB × 5) quando LOG_DIR está setada (env de prod
no Dockerfile). Frontend api.ts anexa `[req:xxxxxxxx]` ao final de
mensagens de erro. Endpoint `/sysadmin/logs/tail` lê últimas N linhas
do log via UI (sysadmin only).

## Why

Amanhã (2026-06-05) o app entra em uso de produção 24/7 single-owner.
Hoje o audit está cobrindo as ações mutantes que retornam 2xx, mas
quando algo der errado eu preciso conseguir investigar **sem SSH** e
**sem perder o trace** da requisição original.

Lacunas atuais:

1. **Erros 4xx/5xx não vão pro audit_log** — `AuditMiddleware.dispatch`
   filtra em `if not (200 <= response.status_code < 300)`. POST que
   estourou 500 fica só no stdout do container.
2. **Sem request ID / correlation ID** — não dá pra costurar uma
   cadeia de chamadas (request → audit log → traceback no stdout).
3. **Logs do container vão pro stdout** — `docker logs` ainda
   funciona, mas se o container for `docker rm` o histórico vai
   junto. Nada persistido em volume.
4. **Sem endpoint pra ler logs via UI** — pra debug rápido preciso
   abrir SSH ou pull dos arquivos.

## Decisões locked (sugestão recomendada da spec anterior)

| # | Item | Decisão |
|---|---|---|
| 1 | Persistir 5xx no audit_log | **Sim** — não somente 2xx mutating. Loga TODOS os 4xx/5xx independente do método. |
| 2 | Request ID middleware | **Sim** — UUID por request, exposto via header `X-Request-ID` na resposta, embutido no audit_log.details + nos logs Python via filter. |
| 3 | Persistir logs do container em volume | **Sim** — `data/logs/numis.log` + rotação 10MB × 5. Sobrevive `docker rm`. |
| 4 | Endpoint pra ler tail dos logs | **Sim** — `GET /sysadmin/logs/tail?n=200` (sysadmin only). Lê últimas N linhas do arquivo de log. |

## In Scope

### 1. Backend — request ID middleware

`src/numis_geek/api/middleware.py`:

- Classe nova `RequestIDMiddleware` que gera UUID curto (8 chars)
  por request, anexa a `request.state.request_id`, e seta como
  `X-Request-ID` response header.
- Wire-in em `app.py` antes do `AuditMiddleware`.

### 2. Backend — audit de 4xx/5xx

`AuditMiddleware.dispatch` ganha um bloco novo:

- Se status >= 400: loga independente do método (incluindo GET) com
  action `http.<status_class>.<método>.<path>` (e.g.,
  `http.5xx.get./api/users/me`).
- Inclui `request_id` no `details`.
- Inclui `status_code` exato no `details`.

A regra de skip `_is_explicit` permanece pra não duplicar audit
quando uma route explicitly faz seu próprio AuditService.log.

### 3. Backend — logging em arquivo + rotação

`src/numis_geek/logging_config.py` (novo):

```python
def configure_logging(log_dir: Path | None = None) -> None:
    """RotatingFileHandler + StreamHandler simultâneos.

    Em prod: data/logs/numis.log (10MB × 5).
    Em dev (sem LOG_DIR env): só StreamHandler (stdout).
    """
```

Chamado uma única vez no `app.py` lifespan startup.

### 4. Backend — endpoint `/sysadmin/logs/tail`

`src/numis_geek/api/routes/sysadmin_logs.py` (novo):

```python
@router.get("/logs/tail")
def tail_logs(
    n: int = Query(default=200, ge=1, le=2000),
    current_user: UserContext = Depends(get_current_user),
):
    """Lê últimas N linhas de data/logs/numis.log. Sysadmin only."""
```

Não inclui no router master se `LOG_DIR` não estiver set (dev).

### 5. Frontend — surface request ID em error toasts

`api.ts` — quando um request falha, extrai `X-Request-ID` do
response (se presente) e inclui no Error message. Assim quando
algo der ruim você copia o request ID e consegue achar no
audit_log + nos logs.

### 6. Docker — montar volume pra logs

`Dockerfile`: `RUN mkdir -p data/logs` (já tem
`data/attachments data/backups`, só estender).

`deploy.sh` / `docker-compose.yml` — bind mount `./data/logs`
persistente (coordenar com vps-devops na hora do deploy).

### 7. Testes

- `tests/test_observability.py`:
  - request ID middleware seta header.
  - request ID está em `request.state` dentro de route.
  - 500 grava audit_log com action começando em `http.5xx`.
  - 401 grava audit_log com action `http.4xx`.
  - Login bem-sucedido (já gravado explicitamente) NÃO duplica.
  - `/sysadmin/logs/tail` exige sysadmin role.

## Out of Scope

- Sentry / Datadog / log aggregator (pequeno demais pra isso).
- Frontend Sentry boundary. Hoje as exceptions React aparecem só
  no console do browser.
- Audit retention / purge automático.

## Migration / Deploy

- Sem schema change (audit_log já aceita JSON details).
- Backend volta pro topo após deploy normal.
- vps-devops precisa garantir bind mount de `data/logs` no
  docker-compose.yml.
