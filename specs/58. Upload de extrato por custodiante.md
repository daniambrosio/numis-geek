# Spec 58 — Upload de extrato por custodiante

## Status: ✅ Done (2026-06-07) — Stages 1+2 implementados. Stages 3+4 pendentes (próximas specs).

Backend Stage 1: migration `spec58_fi_id` aplicada (extraction_job.institution_id
nullable FK). Service + classify + endpoint per-FI funcionando. 2 testes
novos (scope-by-FI + legacy path). 31 passes em test_bulk_extract.

Frontend Stage 2: PendencyPanel agora tem BulkAttachmentManager DENTRO
de cada FI group (institutionId passado como prop). Upload por FI
auto-dispara extração no endpoint scoped. Modal de review esconde
dropdown de FI quando job é scoped. 14 passes em test_pendency_panel +
test_snapshot_detail.

## Why

Hoje o fluxo de bulk extract é **por snapshot**, não por custodiante:

1. User abre snapshot IN_REVIEW.
2. Faz upload de UM extrato consolidado no painel de pendências.
3. LLM extrai todas as posições (genérico, sem saber a corretora).
4. No review modal, user escolhe FI no dropdown.
5. Backend casa o extrato com TODOS os ativos da workspace (filtra por FI só depois).

Problemas:

- **Pool grande no matcher**: 154 ativos da workspace competem todos.
  Treasury americano `T 4 02/15/34` da Avenue compete com Tesouro IPCA
  da XP. Gera ambiguidades, força heurísticas paranoicas, gera falsos
  negativos (Spec 57 follow-ups).
- **Prompt LLM genérico**: tenta cobrir qualquer corretora. Resultado:
  funciona pra ações simples, falha em renda fixa / fundos
  exclusivos / treasuries. Cada corretora formata diferente.
- **UX confusa**: user precisa lembrar de selecionar a FI no review.
  Esquecer ou errar → matching contra workspace inteira → órfãs falsas.

Referência: o código legado (`igeek/extrato/extrato.py`) tinha
**roteamento por corretora** (XP XLSX → `parse_xp_posicao`,
FolhaPrev → `parse_folhaprev`, imagem → LLM minimal pra saldo
único). Funcionava em uma rodada porque a corretora era conhecida
antes de chamar o LLM.

## Decisões locked

| # | Pergunta | Resposta |
|---|---|---|
| 1 | Escopo do upload | **Por FI/custodiante**. Cada FI group no painel tem seu próprio dropzone + lista de anexos. |
| 2 | Upload semântico | One-step: drop file → upload + auto-trigger extraction com `institution_id=this FI`. Sem "Extrair" separado pra V1. |
| 3 | Re-extrair | Botão "Re-extrair" no anexo dispara novo job (mesma FI). |
| 4 | Atribuição da FI | `extraction_job.institution_id` (nullable FK). Attachment table não muda. |
| 5 | Matching scope | `_classify_bulk_extract` filtra `Asset.account.financial_institution_id == job.institution_id` quando set. |
| 6 | Modal de review | Quando job tem `institution_id`, esconde dropdown de FI (já scoped). |
| 7 | Anexos legados | Jobs sem `institution_id` mantêm comportamento atual (`institution_short_name` do body). |
| 8 | "Sem instituição" group | Mantém o BulkAttachmentManager sem FI scope (fallback). |
| 9 | Templates LLM por broker | **Out of scope**. Próxima spec (Stage 3). |
| 10 | Parser XLSX dedicado | **Out of scope**. Próxima spec. |

## In Scope (V1 — Stages 1+2)

### Stage 1 — Backend

**1.1 Migration**

```
extraction_job.institution_id  String(36)  nullable=True
  FK → financial_institution.id  ondelete=SET NULL
```

Sem backfill — jobs antigos ficam com `NULL`, preservando comportamento
atual (review modal mostra dropdown de FI). Reversível.

**1.2 Modelo**

`ExtractionJob.institution_id: Mapped[str | None]` — adicionado.

**1.3 Service**

`create_and_run` aceita `institution_id: str | None = None`,
propaga pro `ExtractionJob`.

`_classify_bulk_extract` resolve a FI:

- Se `job.institution_id` está set → carrega `FinancialInstitution`,
  usa o `short_name` pro display E filtra `Asset` candidates pela
  conta de mesma FI (true scoping, não só name comparison no fim).
- Se não → usa `institution_short_name` do body (caminho legado).

`preview_bulk_extract` idem (já delega pra `_classify_bulk_extract`).

**1.4 Rotas novas**

```
POST /snapshots/{snapshot_id}/institutions/{fi_id}/bulk-extract
  body: { attachment_id: str }
  → BulkExtractJobOut

GET /snapshots/{snapshot_id}/institutions/{fi_id}/extractions
  → list[BulkExtractionJobSummary]   (filtered by job.institution_id)
```

Endpoints antigos (`POST /snapshots/{snap}/bulk-extract`,
`GET /snapshots/{snap}/extractions`) permanecem — caminho "sem
instituição" pro bucket residual.

### Stage 2 — Frontend

**2.1 `BulkAttachmentManager` aceita prop `institutionId?: string | null`**

- Quando set: upload usa endpoint per-FI; lista filtra jobs por
  `institution_id === institutionId`.
- Quando null/undefined: comportamento atual (per-snapshot).
- Auto-trigger de extraction após upload bem-sucedido (one-step UX).

**2.2 `PendencyPanel`: manager DENTRO de cada FI group**

Em vez de UM manager acima de todos os grupos, cada FI tem o seu:

```
▸ Avenue (23 abertas)
   [📎 Anexos / Subir extrato — drag, click, paste]
   ├── extrato-avenue-mai26.pdf · EXTRAÍDO  [Revisar] [Re-extrair]
   └── ações XYZ
   [lista de pendências da Avenue]

▸ XP Investimentos (45)
   [📎 Anexos / Subir extrato]
   ...
```

Grupo "Sem instituição" mantém manager sem FI scope (legado).

**2.3 `BulkExtractReviewModal`: esconde dropdown quando scoped**

Se o job tem `institution_id` (vinda na response), `fiShortName` é fixo
e o dropdown é substituído por um label "Extrato de: Avenue".

## Out of Scope (próximas specs)

- Templates LLM específicos por broker (Stage 3).
- Parsers determinísticos XLSX/CSV por broker (Stage 4).
- Reatribuir FI de anexo já existente.
- Multi-FI num único arquivo — user crops/separa.
- Per-FI auto-priced skip otimização (já existe globalmente).

## Risks / Mitigations

| Risco | Mitigação |
|---|---|
| Anexos uploadados antes da spec ficarem "órfãos de FI" | Bucket "Sem instituição" mantém comportamento atual. User pode re-extrair via grupo "Sem instituição" ou deletar+re-uploadar em FI específica. |
| Asset sem `account_id` ou `account` sem `financial_institution_id` | Filtro escopado por FK pula esses (não entram no pool). Hoje só ativos manuais sem conta → não esperados em extrato bulk. |
| One-step upload + auto-extract gasta tokens à toa se user errar arquivo | Aceita custo (~$0.01 por call). User pode deletar anexo e refazer. Alternativa: manter botão "Extrair" separado se ficar caro. |
| `institution_id` no job adiciona FK constraint frágil | `ON DELETE SET NULL` — deletar FI não derruba job histórico. |
