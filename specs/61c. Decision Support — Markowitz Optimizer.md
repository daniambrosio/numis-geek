# Spec 61c — Decision Support: Markowitz Optimizer

## Status: 📝 Draft — aguarda entrevista própria antes da implementação

## Priority

Alta — peça central do Decision Support v1 (seções (a) Gap vs Target +
(b) Markowitz na página integrada).

## Why

User quer recomendação quantitativa de rebalance baseada em Modern
Portfolio Theory: dados os retornos históricos e correlações do portfólio
atual, mais as metas por classe e caps, encontrar a alocação de menor
variância. Resultado: lista de trades (vender X, comprar Y) pra atingir
o portfólio "ótimo".

## Scope (proposto, revisa na entrevista)

### Backend

- Dependências novas em `pyproject.toml`: `numpy>=1.26`, `scipy>=1.11`.
- `services/markowitz.py`:
  - `build_monthly_returns(db, workspace_id, *, min_months)` — monta
    matriz de retornos mensais a partir de `PortfolioSnapshotItem`.
    Pra cotados: usa `unit_price` mês a mês. Pra modo-valor
    (FUND/PREV/FGTS/CASH): usa
    `(gross_t − gross_{t-1} − aportes_líquidos) / gross_{t-1}`
    derivado de `AssetMovement` no intervalo (snapshot_{t-1},
    snapshot_t]. Multi-moeda: tudo convertido pra BRL antes (aceita
    ruído cambial).
  - `covariance_matrix(returns)` — calcula Σ. Aplica Ledoit-Wolf
    shrinkage `Σ_shrunk = (1−α)·Σ + α·μ·I` com α=0.05 default.
  - `solve_min_variance(...)` — `scipy.optimize.minimize(method='SLSQP')`
    com constraints: `Σw=1`, equality por classe (= class_targets),
    inequality por país (≤ country_caps), bounds `0 ≤ w_i ≤ asset_cap`.
  - `build_frontier(...)` — 20 pontos parametrizados por target_return.
  - `optimize_portfolio(db, MarkowitzInput)` — entry point.
- Detecção de infactibilidade pre-solver (ex: classe REIT target 30% com
  só 1 REIT existente e cap 15% → infeasible) → erro 422 com mensagem.
- Endpoint `POST /api/portfolio/optimize`:
  - Body: `{class_targets, asset_cap=0.15, country_caps={BR:0.70}, workspace_id?}`
  - Retorno: `MarkowitzResult` (frontier, optimal weights vs current,
    trades, binding constraints, excluded assets, warnings).
- Sem persistência v1 — recomputa por demanda.

### Frontend

- `pages/DecisionSupport.tsx`:
  - Seção (a) Gap vs Target — `<GapVsTargetChart>` SVG: barras
    horizontais sobrepostas (atual + outline alvo) por classe e por país.
  - Seção (b) Markowitz — `<OptimizationControls>` (botão Otimizar +
    edit class targets inline) → `<EfficientFrontierChart>` SVG (vol×ret,
    20 pontos da fronteira + marcador current + marcador optimal) +
    `<SuggestedTradesTable>` (`ticker | classe | atual% | alvo% | Δ R$ |
    ação`).
- API client: `api.optimizePortfolio(input)`.
- Gráficos: **SVG puro** (não adicionar recharts/visx — mantém padrão
  do projeto).

## Open Questions (entrevistar)

- **Ledoit-Wolf default vs opt-in:** ligar α=0.05 sempre, ou expor
  toggle pra user? Recomendação: default on, sem toggle v1.
- **Target retorno custom na UI:** mostrar input "quero X% a.a."?
  Decisão atual foi rejeitar Sharpe — mas target retorno pode ser útil
  pra deslizar pela fronteira. Default: omitir v1; recomputa min-variance
  com as class_targets como única âncora.
- **Ativos com <12 meses de histórico:** excluir do solver e listar como
  "excluído por falta de histórico" — confirmar comportamento.
- **Infactibilidade:** devolver 422 com mensagem clara ("Classe REIT
  precisa de ≥ 2 ativos") ou relaxar constraints automaticamente? V1:
  422 explícito.
- **Multi-moeda:** v1 converte tudo pra BRL antes (ruído cambial entra
  como risk). Confirmar tradeoff aceito.
- **Cap por sector:** rationale menciona — fica fora v1 (depende de
  `Asset.sector`).
- **Custom country cap (não só BR):** UI pra editar caps por país
  arbitrário? Default v1: cap fixo BR=70% (já vem do form).
- **Snapshot mínimo:** se workspace tem <12 snapshots, página exibe
  empty state explicativo. Confirmar UX.

## Decisões locked (do master spec 61)

- Frequência mensal v1 (diária em v2 quando spec 53 entrar).
- Universo: TODAS as classes (cotado + modo-valor calculado).
- Objetivo: min-variance com class_targets como equality. Sem Sharpe na UI.
- Restrições: no-short, cap 15%/ativo, cap 70% BR, target_allocation
  como equality por classe.
- Multi-moeda: BRL como unit.
- Persistência: NÃO v1 (recomputa).

## Out of Scope (defer)

- Black-Litterman (priors de mercado).
- CVaR / shortfall optimization.
- Robust portfolio optimization (Tütüncü-Koenig).
- Multi-objetivo (Pareto frontier por objetivos heterogêneos).
- Tax-aware trading (minimize tax impact dos trades).
- Backtesting "se eu tivesse seguido sugestões".
