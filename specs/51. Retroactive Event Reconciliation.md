## Why

Fechamentos são frozen por design (Spec 14/35 — historical truth com
PTAX, fx_rate e items cravados na criação). Mas o usuário aprende
sobre eventos passados depois — corretora atrasa nota de corretagem,
gestora demora a divulgar split, FUND mensal só publica saldo no mês
seguinte. Resultado: ao registrar o evento, **o histórico no app
diverge da realidade** e o snapshot fica errado.

Hoje o sync (`POST /snapshots/{id}/sync-items`, Spec 49 hotfix #12.1)
só faz **add missing** — não detecta mudança de qty/basis em items
existentes. Reabrir + recompute também não é suficiente porque o
reopen mantém items frozen.

A proposta: **reconciliar pelo evento gerador, não pelo snapshot**.
Quando o usuário cadastra uma movimentação/corp action retroativa, o
sistema detecta quais fechamentos seriam afetados, mostra o impacto
detalhado, e aplica sob aprovação explícita.

Volume esperado: ~3-4 ajustes/ano em prod (~realistic, baseado em
splits, atrasos de nota de corretagem), mas **alto nos próximos dias**
enquanto a base de produção ainda está sendo carregada.

## Decisões locked (de entrevista 2026-06-04)

| # | Pergunta | Resposta |
|---|---|---|
| 1 | Reabrir CLOSED automaticamente após confirm? | **Sim** — automático com `reason="recompute por <event_type> <event_id>"`. Snapshot vai de CLOSED → IN_REVIEW → (recompute) → CLOSED. |
| 2 | Trail no item ou só audit_log? | **Só audit_log global.** Sem migration; aceita perda de visibilidade na UI do item. |
| 3 | Se user cancela o prompt, gravar nota de "drift consciente"? | **Não.** Cancelamento pode ser acidental (ESC sem querer) ou deliberado pra avaliar depois. O evento gerador ganha botão "Verificar impacto" pra trazer o prompt de volta. Só se o user explicitamente clicar "Manter divergência" no segundo prompt é que vira nota. |
| 4 | Distribution retroativa: prompt ou silently? | **Silently.** Não muda items dos snapshots, só agregados de proventos. |
| 5 | CorporateAction afeta todos snapshots ≥ event_date. Aplicar em lote? | **Sim**, um prompt único com lista expandida. |
| 6 | Frequência? | 3-4/ano em prod; vários/dia em dev nas próximas semanas. |

## Implicações da decisão #2 (audit-only)

- **Sem migration de schema** — mais rápido implementar.
- **PRÓ:** `AuditLog` já existe e tem (`resource_type`, `resource_id`),
  então filtrar histórico de um item é uma query simples.
- **CONTRA — visibilidade:** a UI do snapshot item NÃO mostra
  "recomputado em X por Y" diretamente; precisaria abrir o painel de
  audit pra ver. Por agora aceitável (volume baixo, user só verifica
  quando desconfia).
- **CONTRA — durabilidade:** audit_log é mais facilmente truncável
  (backup, GDPR-style purge no futuro). Se virar problema, migration
  pra `PortfolioSnapshotItem.recompute_notes JSON` resolve.
- **CONTRA — "drift consciente":** sem campo dedicado, o registro de
  "user optou por manter divergência" também fica no audit_log. UI não
  mostra um pill "drift" no item. Aceitável dado o volume.

## In Scope

### 1. Backend

**1.1 Service — `find_affected_snapshots()` em `services/snapshot.py`**

```python
@dataclass
class AffectedSnapshot:
    snapshot_id: str
    ym: str                       # "2026-04"
    status: SnapshotStatus
    has_item: bool                # já existe item pro ativo?
    old_quantity: Decimal
    new_quantity: Decimal
    old_market_value_brl: Decimal | None
    new_market_value_brl: Decimal | None
    old_total_invested_brl: Decimal | None
    new_total_invested_brl: Decimal | None


def find_affected_snapshots(
    db: Session,
    *,
    workspace_id: str,
    asset_id: str,
    earliest_event_date: date,
) -> list[AffectedSnapshot]:
    """Lista os snapshots cujos items do ativo seriam alterados pelo
    estado atual de movimentos + corp actions, comparado ao que está
    frozen."""
```

Implementação: itera `snapshots WHERE workspace_id = ? AND period_end_date >= ?`,
roda `compute_position(asset_id, as_of=period_end)`, compara com item
existente. Retorna só os que mudariam (qty OU basis OU market_value).

**1.2 Service — `apply_recompute_to_snapshot()`**

```python
def apply_recompute_to_snapshot(
    db: Session,
    *,
    snapshot_id: str,
    asset_id: str,
    trigger_event_type: str,    # "asset_movement.create", etc.
    trigger_event_id: str,
    user_id: str | None,
    user_email: str | None = None,
) -> PortfolioSnapshotItem:
    """Recomputa o item, respeitando fx_rate cravado no snapshot.
    Auto-reopen quando CLOSED → IN_REVIEW. Audit log detalhado."""
```

Comportamento:
- Se snapshot está CLOSED → chama `reopen_snapshot()` com
  `reason="recompute por {trigger_event_type} {trigger_event_id}"`.
- Recomputa o item via `compute_position(asset_id, as_of=period_end)`.
- Usa `snap.fx_rate_usd_brl` cravado (não busca novo).
- Atualiza qty, unit_price, market_value_native/brl/usd,
  total_invested_brl.
- Re-detecta pendência pro ativo no novo estado.
- Audit log com **delta detalhado**:
  ```json
  {
    "snapshot_id": "...",
    "asset_id": "...",
    "trigger_event_type": "asset_movement.create",
    "trigger_event_id": "mov-uuid",
    "before": {"quantity": "100", "market_value_brl": "5000"},
    "after":  {"quantity": "200", "market_value_brl": "10000"},
    "auto_reopened": true
  }
  ```
- NÃO fecha de volta automaticamente — user revisita pendências e
  confirma manualmente.

**1.3 Service — `apply_skip_recompute()`** (drift consciente)

```python
def apply_skip_recompute(
    db: Session, *,
    snapshot_id: str, asset_id: str,
    trigger_event_type: str, trigger_event_id: str,
    reason: str,
    user_id: str | None, user_email: str | None = None,
) -> None:
    """Grava no audit_log que o usuário optou por NÃO recomputar
    apesar do drift detectado. Útil pra reconstituir 'por que esse
    item não bate com a verdade atual?'."""
```

Audit action: `snapshot.recompute.skipped`. Sem mutação de dados.

**1.4 Endpoints — em `routes/snapshots.py`**

- `POST /assets/{asset_id}/affected-snapshots`
  - Body: `{ event_date: date }`
  - Não requer evento persistido — pode ser preview "se eu lançar isso, o que afeta?".
  - Útil pra integrar no MovementComposer (preview antes do save).

- `POST /snapshots/{snapshot_id}/items/{asset_id}/recompute`
  - Body: `{ trigger_event_type: str, trigger_event_id: str }`
  - Aciona `apply_recompute_to_snapshot()`.
  - Retorna item atualizado.

- `POST /snapshots/{snapshot_id}/items/{asset_id}/skip-recompute`
  - Body: `{ trigger_event_type, trigger_event_id, reason }`
  - Aciona `apply_skip_recompute()`.

### 2. Eventos cobertos

| Categoria | Entidade | Operação | Tratamento |
|---|---|---|---|
| A | `AssetMovement` | CREATE/UPDATE | Prompt obrigatório |
| A | `AssetMovement` | DEACTIVATE | Prompt obrigatório (reverte delta) |
| A | `CorporateAction` SPLIT/GROUPING | CREATE/UPDATE | Prompt obrigatório (afeta MUITOS snapshots) |
| A | `CorporateAction` ASSET_CONVERSION | CREATE | Prompt obrigatório (afeta 2 ativos × N snapshots) |
| B | `Distribution` | CREATE/UPDATE/DEACTIVATE | Silently — não muda items |
| C | `Asset` metadata, `current_price` | Qualquer | Não afeta — fora de scope |

### 3. Frontend

**3.1 `<AffectedSnapshotsModal />` — modal reusável**

Disparado por:
- Após `onSave` em `MovementComposer` (passando movement_id).
- Após save em `OptionModal` quando criar movement.
- Após save em formulário de CorporateAction (Spec 13 — quando
  implementado/revisto).
- Botão "Verificar impacto em fechamentos" no detail panel da
  movimentação/CA (segunda chance).

Props:
```ts
interface Props {
  assetId: string
  eventDate: string          // YYYY-MM-DD
  triggerEventType: string   // "asset_movement.create", etc.
  triggerEventId: string
  onClose: () => void        // ESC / Cancel — sem nota
  onApplied: () => void      // confirmou Aplicar
  onSkipped: () => void      // confirmou Manter divergência
}
```

Layout:
```
┌─ Impacto em fechamentos ────────────────────────────────┐
│                                                          │
│ Esse lançamento afeta 3 fechamentos:                     │
│                                                          │
│ ☑ Mar/26 (CLOSED)  qty 100→200  R$ 5.000→R$ 10.000      │
│ ☑ Abr/26 (CLOSED)  qty 100→200  R$ 5.100→R$ 10.200      │
│ ☑ Mai/26 (IN_REVIEW) qty 100→200 R$ 5.300→R$ 10.600     │
│                                                          │
│ Os CLOSED serão reabertos automaticamente.               │
│                                                          │
│  [ Cancelar ]  [ Manter divergência ]  [ Aplicar ]      │
└──────────────────────────────────────────────────────────┘
```

Comportamento:
- **Cancelar** (ESC, X, botão): fecha sem efeito. Nenhuma nota gravada.
  O movement/CA continua persistido com o drift latente.
- **Manter divergência**: confirma explicitamente. Abre um campo
  textual obrigatório "Por quê?" → grava `apply_skip_recompute()`
  para cada snapshot listado.
- **Aplicar**: roda `apply_recompute_to_snapshot()` em cada snapshot
  marcado, em sequência. Mostra spinner + count.

**3.2 Botão "Verificar impacto" no detail panel**

Em `LancamentoDetailPanel.tsx` e equivalentes pra CorporateAction:
- Botão "Verificar impacto em fechamentos" visível sempre.
- Click → chama `POST /assets/{asset_id}/affected-snapshots` com
  `event_date = movement.event_date`.
- Se houver afetados, abre `<AffectedSnapshotsModal />`.
- Se zero afetados, mostra toast "Fechamentos já estão sincronizados".

**3.3 Pluga no `MovementComposer.onSave`**

```tsx
const saved = await api.createAssetMovement(payload)
const affected = await api.previewAffectedSnapshots(saved.asset_id, saved.event_date)
if (affected.length > 0) {
  openAffectedSnapshotsModal({
    assetId: saved.asset_id,
    eventDate: saved.event_date,
    triggerEventType: 'asset_movement.create',
    triggerEventId: saved.id,
  })
}
```

## Out of Scope

- **Cascade automático** entre snapshots quando muda corp action —
  spec mantém approval por evento.
- **Atualização visual em outras telas** (Portfolio, Dashboard) durante
  o flow — usuário reabre a aba pra ver.
- **Distribution flow detalhado** — silently update por enquanto.
  Pode virar follow-up quando o impacto começar a doer.
- **Painel "drifts conhecidos"** centralizado — pode aparecer depois
  como filtro no audit_log.
- **Migration pra `recompute_notes` no item** — fica como fallback se
  audit-only se mostrar limitante.

## Acceptance Criteria

1. Salvar movement com `event_date` em mês de snapshot CLOSED dispara
   o modal de impacto.
2. Confirmar "Aplicar" recomputa item + reabre CLOSED automaticamente
   com reason rastreável.
3. Cancelar (ESC, X, "Cancelar") não grava nada.
4. "Manter divergência" exige reason e grava no audit_log.
5. Botão "Verificar impacto" no detail do movement traz o mesmo modal
   posteriormente.
6. SPLIT criado em mês X gera prompt com TODOS snapshots ≥ X.
7. `pytest -q` + `npm run test:run` + `npm run build` verdes.
