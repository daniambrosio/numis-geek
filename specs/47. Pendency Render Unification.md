# Spec 47 — Pendency Render Unification

## Status: ✅ Implementing

## Why

`PendencyPanel.tsx` foi criado no Phase C da Spec 35 (commit `ba0fc1f`) como
componente único de revisão de pendências, conforme o próprio Spec 35
desenha (linhas 392, 485 — "SnapshotDetail.tsx + `<PendencyPanel>`"). Mas
a página `SnapshotDetail.tsx` foi de fato implementada com um
`PendencyRow` **inline** + bloco de listagem próprio. O `PendencyPanel`
ficou **órfão no repositório**: exportado, com testes verdes, mas
ninguém importa.

Sintomas:
- Spec 38 (Upload Extraction via LLM) mexeu no `PendencyPanel` órfão
  achando que era o lugar de produção — resultado: o modal de upload
  real (`ExtractionUploadModal`) **não chega à UI**. O caminho em
  produção usa um `window.prompt` stub.
- Spec 35 hotfix #2 (commit `92f4d8a`) caiu no mesmo armadilha — o
  user reportou que a tela não estava agrupando por FI.
- A correção (`6dec0ad`) replicou a lógica no `SnapshotDetail`,
  fixando o sintoma mas deixando a duplicação no lugar pra próxima
  vez.

## In Scope

1. **`PendencyPanel.tsx` vira o componente de produção.** Adota o
   visual e a API do bloco inline atual (validado pelo user na tela
   de 31/05/2026):
   - Header com `AlertTriangle` + "Resolver pendências antes de fechar"
     + frase descritiva `{pendingTotal} de {totalAssetsCount} ativos...`
   - Botão "Confirmar fechamento" emerald no header, desabilitado
     quando `pendingTotal > 0`
   - Progress bar `{resolvedAssets} de {totalAssetsCount}`
   - Cor binária da barra do ativo (âmbar=manual, vermelho=API)
   - Footer "Como o sistema processa: upload de PDF/screenshot/CSV/XLSX..."
   - Grouping por instituição (carregado do hotfix #2)
   - Linha mostra `· {Mês}/{ano}: R$ X,XX` quando há preço prévio
     + botão **Repetir** (indigo)
   - Botão **Editar** com prompt pré-preenchido
   - Botão **Upload extrato** abre `ExtractionUploadModal` (Spec 38
     real, recuperado do componente órfão)

2. **`SnapshotDetail.tsx` importa `<PendencyPanel>`** e deleta:
   - `function PendencyRow(...)` inline
   - `groupPendenciesByFI` (move pro PendencyPanel)
   - `UNGROUPED_FI_LABEL` (idem)

3. **Props que `PendencyPanel` precisa receber:**
   ```ts
   {
     pendencies: SnapshotPendencyOut[]
     assetById: Map<string, AssetOut>      // pra ClassBadge
     pendingTotal: number
     totalAssetsCount: number
     resolvedAssets: number
     periodEndDate: string                  // pra header
     onResolved: () => void
     onConfirm?: () => void
   }
   ```

## Out of Scope

- Mudar comportamento de auditoria, endpoints ou shape de
  `SnapshotPendencyOut` — só refactor de UI.
- Tocar nos testes do componente órfão (que rodavam, mas testavam
  algo que não chega ao usuário). Substituir por testes da nova API.

## Acceptance Criteria

1. `SnapshotDetail.tsx` não tem `function PendencyRow` nem
   `groupPendenciesByFI`. Importa e renderiza `<PendencyPanel ... />`.
2. `PendencyPanel.tsx` é o único renderer de pendências no app.
3. Botão "Upload extrato" abre `ExtractionUploadModal` (Spec 38 real),
   não `window.prompt`.
4. Visual idêntico ao da tela validada em 31/05/2026 (header +
   progress + grouping + Repetir + valor anterior + footer).
5. `npm run test:run` verde; `npm run build` clean; `pytest -q` verde.

## Rationale: por que não fizemos isso antes

Quando o Spec 35 hotfix #2 foi implementado, o atalho foi reproduzir
a lógica nova no `SnapshotDetail` em vez de unificar — entregava o
fix pro user fechar Maio no mesmo dia, mas deixava a dívida no lugar.
Esse spec paga a dívida agora, antes que próximo hotfix de pendência
caia no arquivo errado de novo.
