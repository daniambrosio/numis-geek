# Spec 53 — Historical Price Fetch

## Status: 📋 Planning (não implementada ainda)

## Why

Spec 52 corrigiu o leak de preço LIVE em snapshots antigos: quando o
sistema precisa de um preço pra um period_end histórico, cria uma
`SnapshotPendency` `HISTORICAL_PRICE_REQUIRED` pra você preencher
manualmente.

Isso resolve a corrupção mas adiciona fricção. Cenários comuns:

- Mov retroativa que introduz asset novo num snapshot: hoje pendency
  manual.
- Sync de asset que faltava em snapshot antigo: hoje pendency manual.
- Retry API em pendency de snapshot antigo: hoje bloqueado, força
  EDIT_PRICE manual.

Os providers que já usamos (brapi, finnhub, coinbase, yfinance) **têm
endpoints históricos**. Esta spec integra esses endpoints como uma
camada de auto-preenchimento das pendências `HISTORICAL_PRICE_REQUIRED`
e como fonte legítima do `retry_pendency_api` em snapshot antigo.

Resultado esperado: em ~90% dos casos retroativos, o user não precisa
digitar nada.

## Decisões pendentes (a discutir antes de implementar)

| # | Pergunta | Tendência |
|---|---|---|
| 1 | Tentar fetch histórico ANTES de criar pendency, ou criar pendency primeiro e oferecer "preencher via API" no UI? | **Antes** — fricção zero quando o provider responde. Pendency só persiste se o fetch falhar. |
| 2 | Dia não-útil — period_end ~ último dia do mês pode cair sábado/domingo. Provider responde "no data". Backwalk pro último dia útil? | **Sim, backwalk até 3 dias úteis pra trás.** Match com a lógica do PTAX walkback (já existente). |
| 3 | Cache local de preços históricos buscados? | **Não** — `PortfolioSnapshotItem.unit_price` já é o cache. Mesmo asset em mesmo period_end nunca dispara fetch duas vezes. |
| 4 | API rate limit. Spec 51 pode disparar fetch em N snapshots simultaneamente (split atrasado). | **Sim, com paralelismo limitado (3 concorrentes) + retry exponencial.** Mesmo padrão da bulk refresh. |
| 5 | Asset delistado / provider não retorna histórico (BR small caps em yfinance, p.ex.). | **Fallback chain:** brapi historical → yfinance historical → pendency manual. Para US: finnhub → yfinance → pendency. |
| 6 | Atualizar `retry_pendency_api` pra usar histórico em snapshot antigo? | **Sim** — é o equivalente histórico do refresh_one. |

## Escopo

### 1. Adapter histórico por provider

Cada integration ganha função `fetch_quote_on(symbol, token, target_date) -> Decimal`:

| Provider | Endpoint | Notas |
|---|---|---|
| brapi | `quote/{ticker}/historical-data` ou `quote/{ticker}?range=...&interval=1d` | Token obrigatório. Walkback se data não existe. |
| finnhub | `/stock/candle?symbol=...&from=...&to=...&resolution=D` | Token obrigatório. Splits aplicados pelo Finnhub? Confirmar. |
| coinbase | `/products/{pair}/candles?start=...&end=...&granularity=86400` | Sem token. Par BTC-USD, USDC-USD etc. |
| yfinance | `yfinance.Ticker(...).history(start, end)` | Sem token; lib Python. Mais resiliente pra BR small caps. |

### 2. Service novo: `services/historical_price.py`

```python
def fetch_on(db: Session, asset: Asset, target_date: date) -> Decimal | None:
    """Tenta buscar o preço de fechamento de target_date pro asset.

    Dispatch idêntico ao _fetch_price em price_update.py.
    Faz walkback até 3 dias úteis pra trás se o provider responder
    "no data" pro target exato.
    Retorna None se todos os providers da chain falharem.
    """
```

Wire-in nos paths da Spec 52:

- `_new_item_price_or_pendency` tenta `fetch_on(period_end)` primeiro.
  Sucesso → escreve `unit_price`. Falha → mantém pendency
  `HISTORICAL_PRICE_REQUIRED`.
- `retry_pendency_api` em snapshot antigo: chama `fetch_on(period_end)`
  em vez de `refresh_one`. Sucesso → grava + resolve pendency.

### 3. Testes

- Mocks de cada adapter (brapi/finnhub/coinbase/yfinance) retornando:
  preço, no-data, erro de rede, rate-limit.
- Walkback: target=sábado, mock retorna no-data pra sábado e sexta,
  então retorna preço pra quinta. Verifica que `fetch_on` retorna o
  preço da quinta.
- Cenário E2E: criar mov retroativa em snapshot 3 meses atrás de asset
  novo; assert `unit_price` veio do brapi historical mock + 0
  pendências criadas.
- Fallback chain: brapi mock → erro 500. yfinance mock → preço OK.
  Verifica unit_price veio do yfinance.
- Rate limit: 10 snapshots afetados em paralelo → max 3 fetches
  simultâneos.

### 4. Tokens

Reusa `IntegrationCredential` (Spec 23). Nenhum novo secret.

### 5. Edge cases conhecidos

- **End-of-month em fim de semana:** walkback até 3 úteis pra trás.
- **Cripto sem par USD:** coinbase falha → yfinance pode ter.
- **Asset delistado:** todos os providers falham → mantém pendency
  manual (user preenche com referência).
- **Símbolo errado/cadastro inconsistente:** mesmo fallback de
  refresh_one — pendency com mensagem clara.

### 6. Métricas

- Audit log `price.fetch_historical` por provider + status (ok/no-data/error).
- Permite avaliar taxa de sucesso (esperado ≥80% pra BR liquid + 90%
  pra US).

## Dependências

- Spec 52 entregue (esqueleto da pendency + helper
  `_new_item_price_or_pendency`).
- Sem mudanças no `IntegrationCredential` — já temos tokens.
- Sem mudança no model de PendencyReason — `HISTORICAL_PRICE_REQUIRED`
  continua existindo como fallback.

## Out of Scope

- UI de "ver dados históricos do provider" pré-confirmação. Auto-preencher
  é suficiente; user vê só o resultado.
- Re-fetch periódico de preços históricos pra "corrigir" valores manuais
  digitados pelo user. Manual > automatizado.
- Múltiplas tentativas de provider primário sem fallback. Fallback chain
  é o melhor que conseguimos sem complicar UX.
