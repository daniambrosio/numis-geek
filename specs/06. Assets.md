# Spec 06 — Assets (Ativos)

## Status: 📝 Planned

## Overview

Assets represent everything an investor or owner can hold value in: tradable securities (Brazilian and US stocks, FIIs, ETFs, REITs, Bonds, Funds, Crypto), fixed-income instruments (CDB, Tesouro Direto, LCI, LCA), and physical patrimony (real estate, vehicles).

The asset catalog is **workspace-scoped** — each workspace maintains its own assets — but `sysadmin` can view and manage assets across all workspaces (consistent with the project's `Roles & Authorization` policy).

Each asset is **bound to a Financial Institution as custodian** — the place where the asset is held. The same security held at two different brokers is modeled as **two separate asset rows** (e.g., `PETR4 @ XP` and `PETR4 @ Itaú` are distinct). For physical patrimony or assets without a real custodian, a system-level "Particular" institution is used.

This spec creates the foundation for **Lançamentos** (entries: purchases, sales, dividends, come-cotas, etc.). It does **not** include current price / quote management — that is a separate future spec (roadmap feature 19).

## Model: `asset` (core)

| Column | Type | Notes |
|---|---|---|
| id | UUID PK | |
| workspace_id | UUID FK → workspace | **NOT NULL**, **immutable** — every asset belongs to one workspace and cannot be moved |
| financial_institution_id | UUID FK → financial_institution | **NOT NULL** — custodian (where the asset is held) |
| asset_class | enum `AssetClass` | NOT NULL |
| subtype | str(100) \| None | free-text categorization |
| name | str(255) | e.g., "Petrobras PN", "Apto Pinheiros 302" |
| ticker | str(20) \| None | required for ticker-based classes (see Validation) |
| cnpj | str(18) \| None | only for `FUND` (BR fund identifier) |
| currency | enum `Currency` (`BRL`/`USD`) | NOT NULL |
| notes | text \| None | free-text notes per asset |
| is_active | bool | default `True`; soft-delete |
| created_at | datetime | UTC auto |
| updated_at | datetime | UTC auto, onupdate |
| created_by | UUID FK → user | |
| updated_by | UUID FK → user | |

**Indexes / constraints**
- Partial unique index on `(workspace_id, ticker, financial_institution_id)` where `ticker IS NOT NULL` — `asset_class` is intentionally **not** in the key so that a class typo (e.g., creating PETR4 as STOCK_US instead of STOCK_BR at the same custodian) is caught. Same security held at two different custodians is allowed (and expected) as two rows.
- FK to workspace; FK to financial_institution (RESTRICT on delete — IF cannot be deactivated/deleted while assets reference it); FKs to user for `created_by` / `updated_by`.

## Specialized Models (1:1 with `asset`)

For asset classes that need extra structured fields, a sibling table extends `asset` with `asset_id` as both PK and FK. The specialized row is created/updated/deleted in the same transaction as the parent `asset` row.

### `fixed_income_asset`

For `asset_class = FIXED_INCOME`.

| Column | Type | Notes |
|---|---|---|
| asset_id | UUID PK FK → asset | cascade delete |
| issuer | str(255) | e.g., "Banco BTG", "Tesouro Nacional" |
| issue_date | date \| None | data de emissão |
| maturity_date | date | NOT NULL — vencimento |
| indexer | enum `FixedIncomeIndexer` | `CDI`, `IPCA`, `SELIC`, `PREFIXED`, `USD` |
| rate | numeric(8,4) | meaning depends on indexer (e.g., `110.0000` for "110% do CDI"; `6.5000` for "IPCA + 6.5%"; `12.5000` for prefixado 12.5% a.a.) |
| face_value | numeric(15,2) \| None | optional notional / valor nominal |

### `physical_asset`

For `asset_class IN (REAL_ESTATE, VEHICLE)`. Single table with all fields nullable; the application layer enforces which fields are required per class. Generic name chosen because future physical-asset classes (e.g., joias, obras de arte) can reuse it without a new table.

| Column | Type | Notes |
|---|---|---|
| asset_id | UUID PK FK → asset | cascade delete |
| **Real-estate fields** | | required when `asset_class = REAL_ESTATE` |
| address | str(500) \| None | logradouro completo |
| city | str(100) \| None | |
| state | str(50) \| None | |
| country | str(2) \| None | ISO 3166-1 alpha-2 (`BR`, `US`, …) |
| area_m2 | numeric(10,2) \| None | |
| registration_number | str(100) \| None | matrícula / registry number |
| **Vehicle fields** | | required when `asset_class = VEHICLE` |
| make | str(100) \| None | e.g., "Toyota" |
| model | str(100) \| None | e.g., "Corolla" |
| year | int \| None | model year |
| license_plate | str(20) \| None | placa |
| chassis | str(50) \| None | VIN / chassi |

Other classes (STOCK_BR, STOCK_US, FII, ETF, REIT, BOND, FUND, CRYPTO) use only the core `asset` table.

## Validation Rules

Enforced in the API layer (Pydantic models + service layer) and re-checked at DB level where possible.

- `financial_institution_id` is **required** for every asset, regardless of class. For physical patrimony (REAL_ESTATE, VEHICLE) or assets without a real custodian, the user picks the system-level **"Particular"** institution (or the financing bank, exchange, etc., as preferred).
- The chosen `financial_institution` must be active (`is_active = True`).
- `ticker` is **required** when `asset_class IN (STOCK_BR, STOCK_US, FII, ETF, REIT, BOND, CRYPTO)`.
- `ticker` must be **null** when `asset_class IN (FIXED_INCOME, REAL_ESTATE, VEHICLE)`.
- `ticker` is **optional** for `FUND` (some workspaces may want to use a short code).
- `cnpj` is allowed **only** when `asset_class = FUND`.
- The `details` payload (specialized fields) **must** be present when class is `FIXED_INCOME`, `REAL_ESTATE`, or `VEHICLE`; **must** be absent otherwise. For REAL_ESTATE, `address`, `city`, `state`, `country` are required; for VEHICLE, `make`, `model`, `year` are required.
- `currency` defaults (frontend hint, not enforced server-side):
  - `STOCK_BR`, `FII` → BRL
  - `STOCK_US`, `REIT`, `BOND`, `CRYPTO` → USD
  - others: explicit selection required.
- Within a workspace, no two assets with the same `(ticker, financial_institution_id)` when `ticker IS NOT NULL`, regardless of class. The same ticker held at different custodians is allowed.
- `workspace_id` is set on creation and cannot be changed (no asset-move endpoint).

## Permissions

Asset is **workspace-scoped**.

- **member**: read/create/update/deactivate assets in own workspace.
- **admin**: same as member, in own workspace.
- **sysadmin**: read/create/update/deactivate across **all** workspaces. UI exposes a workspace selector. API requests from sysadmin must include a `workspace_id` (query param for list, body field for create) when targeting a specific workspace; omitting it on `GET /assets` returns assets across all workspaces.

Follows the `Roles & Authorization` policy in `CLAUDE.md`: every workspace-scoped query skips the workspace filter when `role == sysadmin`.

## API

| Method | Path | Auth | Description |
|---|---|---|---|
| GET | `/assets` | any authenticated | List assets in caller's workspace. Sysadmin sees all; supports `?workspace_id=`, `?asset_class=`, `?include_inactive=true`, `?search=` |
| GET | `/assets/{id}` | any authenticated | Returns asset core + nested `details` for specialized classes |
| POST | `/assets` | any authenticated | Create asset in caller's workspace (sysadmin may specify `workspace_id` in body) |
| PUT | `/assets/{id}` | any authenticated | Update core fields and `details` (when applicable) |
| PUT | `/assets/{id}/deactivate` | any authenticated | Soft-delete (`is_active = False`) |

**Request/response shape (POST / PUT)**

```json
{
  "asset_class": "FIXED_INCOME",
  "financial_institution_id": "…UUID of XP…",
  "name": "CDB BTG 110% CDI 2028",
  "currency": "BRL",
  "subtype": null,
  "ticker": null,
  "cnpj": null,
  "notes": "Custodiado na XP; emitido pelo BTG",
  "details": {
    "issuer": "Banco BTG Pactual",
    "issue_date": "2024-03-15",
    "maturity_date": "2028-03-15",
    "indexer": "CDI",
    "rate": 110.0,
    "face_value": 50000.00
  }
}
```

Notes:
- `financial_institution_id` represents the **custodian** (where the asset is held), which may differ from the `issuer` of a fixed-income instrument (`issuer` is free-text inside `details` for FIXED_INCOME).
- `workspace_id` is implicit (caller's workspace) for non-sysadmin; sysadmin must include it in the body for `POST` and as a query param for `GET`.
- For ticker classes (e.g., `STOCK_BR`), `details` is omitted entirely.

**Audit actions**: `asset.created`, `asset.updated`, `asset.deactivated`.

## Frontend

- **Routes**
  - `/assets` — admin and member, scoped to current workspace.
  - `/sysadmin/assets` — sysadmin, with workspace selector dropdown at the top.
- **Page files**
  - `frontend/src/pages/Assets.tsx`
  - `frontend/src/pages/sysadmin/Assets.tsx`
- **Table columns**: ticker (or `—`), name, class badge, subtype, currency, updated_at, actions (edit / deactivate).
- **Filters**: class (multi-select), currency, search (matches name + ticker + CNPJ), "show inactive" toggle.
- **Create / edit modal**
  - Class selector is the first field; selecting a class reveals the appropriate sub-form.
  - **Financial Institution selector** (custodian) is required for every class — dropdown of active IFs, with the seeded "Particular" institution as a clear option for personal patrimony.
  - Common fields: name, currency, subtype, notes.
  - Ticker classes: + `ticker`.
  - `FUND`: + `cnpj`, optional `ticker`.
  - `FIXED_INCOME`: + issuer (free-text), issue_date, maturity_date, indexer, rate, face_value.
  - `REAL_ESTATE`: + address, city, state, country, area_m2, registration_number.
  - `VEHICLE`: + make, model, year, license_plate, chassis.
  - `workspace_id` is not user-editable after creation.
- **Sysadmin workspace selector**
  - `/sysadmin/assets` shows a workspace dropdown at the top with "Todos" plus each workspace.
  - "Todos" → list aggregates across workspaces with a workspace column visible.
  - Selecting a specific workspace → list filters to that workspace and the "Create" button pre-fills `workspace_id`.
  - Edit modal opens locked to the asset's workspace (no move).
- **Sidebar**
  - Workspace users: "Ativos" item under the main workspace section.
  - Sysadmin: "Ativos" item appears under the existing **Sistema** section, linking to `/sysadmin/assets`.

## Out of Scope (deferred)

- Current price / quote field on Asset (manual or API-fed) — separate spec.
- Price history table — separate spec.
- Asset valuation methods (Bazin, Graham, Peter Lynch, etc.) — roadmap feature 19.
- Cross-workspace deduplication / shared global catalog — may revisit if duplication becomes painful.
- Linking Asset to a Financial Institution (e.g., CDB tied to a specific bank IF row) — `issuer` is free-text for now; can be tightened later.
- Polymorphic Lançamentos that operate on Assets — that is the next spec.
- Moving an asset between workspaces (transfer / reassignment) — `workspace_id` is immutable in this spec.
- Transferring an asset between custodians (e.g., portability XP → BTG) — represented today by deactivating the old asset and creating a new one; a first-class transfer flow is deferred.
