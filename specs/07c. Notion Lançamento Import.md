# Spec 07c — Notion Lançamento Import

## Status: 📝 Planned

## Overview

One-shot ETL that reads the user's existing Notion database **DB IG Lançamentos** and populates the Numis-Geek `lancamento` table for the "Família Ambrosio" workspace.

Mirror of Spec 07a (asset import) — same pattern: live MCP-driven fetch, intermediate JSON snapshot at `data/notion_lancamento_export.json` (gitignored), idempotent `--apply` against the local DB. Treated as a **utility, not a product feature**: runs once, stops being relevant after the user starts operating in Numis-Geek.

This spec also lands two cross-cutting changes that are foundational for any future import / reconciliation work:

1. **`external_id` + `external_source` columns** on both `asset` and `lancamento` tables (deferred from 07a). Migration retroactively populates Asset's `external_id` from the existing `notes` field.
2. **Position calculator: cost-basis reset** when a sale closes the position (combined explicit flag + numeric tolerance).

A future Spec 07d will import dividends/JCP/JUROS from `DB IG Proventos` (separate Notion DB).

---

## Non-goals (explicitly deferred)

- **DIVIDENDO, JUROS, JCP**: live in `DB IG Proventos` in the user's Notion. Spec 07d.
- **Eventos** (split/inplit corporate actions): live in `DB IG Eventos`. Future spec.
- **Brokerage note import + reconciliation**: this spec adds the `nota_negociacao_number` field on Lançamento so the data is preserved, but the actual "import .pdf de corretagem and link to existing lançamentos" flow is a separate spec.
- **Continuous bidirectional sync**: out of scope. Numis-Geek becomes the source of truth after the import.
- **Aporte / resgate (cash transfers between accounts)**: still deferred (per 07b non-goals).

The Numis-Geek enum gains a 9th value, `RESGATE_TOTAL`. ## Cost basis reset (the "PRIO3 problem")

When a position is fully closed and later re-opened, the new position's PM (preço médio) must **not** include prices from the old, fully-sold position.

### 1. New enum value: `RESGATE_TOTAL`

`LancamentoType` gains a 9th value `RESGATE_TOTAL` (display PT: "Resgate Total"). Validation is identical to `VENDA` (qty + unit_price required for cotado assets; gross_amount-only for não-cotado). The difference is purely semantic + the position-reset behavior in §2.

### 2. Position-calculator update (`services/positions.py`)

The calculator processes lançamentos in chronological order, maintaining `running_qty` and `running_basis`. After processing each sale:

```python
TOLERANCE = Decimal("1e-6")

if lan.type in (VENDA, RESGATE_TOTAL):
    running_qty -= lan.quantity
    if lan.type == RESGATE_TOTAL or abs(running_qty) < TOLERANCE:
        running_qty = Decimal("0")
        running_basis = Decimal("0")  # RESET — next COMPRA starts fresh
```

The tolerance is intentionally **tight** (`1e-6`) — it's only a safety net for Decimal arithmetic edge cases (residuals from bonificação rounding, split factors, fractional-share quirks for crypto). The **`RESGATE_TOTAL` type is the primary, explicit signal**; the tolerance handles the long-tail of mathematical edge cases when the user logs a regular VENDA that mathematically zeros the position.

### 3. Importer behavior

Notion `Tipo Transação = "Resgate Total"` → Numis `LancamentoType.RESGATE_TOTAL`. Direct 1:1 mapping; no transformation logic.

### 4. UI affordance (LancamentoModal)

`RESGATE_TOTAL` shows up as the 9th option in the type dropdown ("Resgate Total"). For users who didn't intend a full close but discovered they sold all units, they pick VENDA — the tolerance auto-resets if it really hits 0.

## Não-Cotado assets: relaxed validation

Notion has two distinct asset shapes:

- **Cotado**: ticker-based (Ações, FIIs, ETFs, REITs, Bonds, Crypto). Rows have `Qtde × Preço Unit.`
- **Não Cotado**: FGTS, CDB / Tesouro / Renda Fixa, Previdência Privada, Dinheiro. Rows have a single `AR Valor` (the operation's monetary value); `Qtde` and `Preço Unit.` are typically empty.

**Validation change** (backend Pydantic + DB constraints):

For `LancamentoType in (COMPRA, VENDA, SUBSCRICAO)`, the requirement becomes:

> Either `(quantity AND unit_price)` are both present, OR `gross_amount` is present standalone. Not both empty.

`BONIFICACAO` still requires `quantity`. Income types (`DIVIDENDO`/`JUROS`/`JCP`/`COME_COTAS`) unchanged.

Concretely: a `COMPRA` of a CDB for R$ 5.000 can be recorded as `gross_amount = 5000, quantity = null, unit_price = null`. The position calculator treats such rows as "+ R$ 5.000 to basis, + 0 to qty". Position reports for non-cotado assets surface "valor aplicado" instead of "qty × PM".

Importer rule:
- If Notion row is `Tipo Ativo = "Não Cotado"` AND `Qtde` is null/0: persist `gross_amount = AR Valor`, leave `quantity` and `unit_price` null.
- Else: persist `quantity = Qtde` and `unit_price = Preço Unit.`; let the route compute `gross_amount`.

## `external_id` + `external_source` (cross-cutting change)

### New columns

Both `asset` and `lancamento` gain:

| Column | Type | Notes |
|---|---|---|
| `external_id` | str(255) \| None | The page URL or unique ID in the source system (e.g., a Notion page UUID, or a broker's nota number) |
| `external_source` | enum `ExternalSource` \| None | Source of truth: `notion`, `b3`, `broker_note`, `manual_csv`, … (start with `notion` only; extend as needed) |

Composite index `(external_source, external_id)` — partial: `WHERE external_id IS NOT NULL`.

### Why both columns

- `external_id` alone is ambiguous across sources (Notion UUIDs may collide with broker note numbers structurally).
- Composite key is the canonical way to identify a row across heterogeneous imports.

### Asset migration (retroactive)

A new Alembic migration:

1. Adds the two columns to `asset` and `lancamento` (NULL allowed, no data backfill needed for `lancamento` since it's empty today).
2. For `asset`: parses the existing `notes` field — if it starts with `"Notion: <url>"`, sets `external_id = <url>` and `external_source = 'notion'`, then strips that prefix from `notes`. If `notes` becomes empty after the strip, set to NULL.
3. Adds the partial composite index.

Round-trip clean: downgrade restores the `notes` prefix from `external_id` so no information is lost.

### Lançamento `nota_negociacao_number`

Separate column on `lancamento` (not on `asset`):

| Column | Type | Notes |
|---|---|---|
| `nota_negociacao_number` | str(50) \| None | The broker note number (`Número Nota Negociação` in Notion). Distinct from `external_id` because a single lançamento can have BOTH a Notion source URL AND a broker note number — and brokerage note imports will need this field as the join key without overwriting the Notion provenance. |

No index on this column for v1 — query patterns are still TBD.

## Architecture (matches 07a)

### Files

- `scripts/fetch_notion_lancamentos.py` — documentation stub (MCP-orchestrated fetch happens from Claude Code session, identical to 07a).
- `scripts/import_notion_lancamentos.py` — JSON → DB. CLI: `--dry-run` (default), `--apply`, `--from-json PATH`.
- `data/notion_lancamento_export.json` — gitignored snapshot.
- `scripts/seed.py` updated — if `data/notion_lancamento_export.json` exists, applies it after assets are seeded (mirrors the 07a hook).

### Snapshot format

```json
{
  "exported_at": "2026-05-04T20:00:00Z",
  "source_database": "DB IG Lançamentos",
  "source_workspace": "daniambrosio.notion.so",
  "target_workspace_name": "Família Ambrosio",
  "lancamentos": [
    {
      "notion_id": "https://www.notion.so/...",
      "asset_external_id": "https://www.notion.so/asset-page-...",
      "type": "COMPRA",
      "event_date": "2024-03-15",
      "quantity": 100.0,
      "unit_price": 30.50,

      "fee": 1.50,

      "currency": "BRL",

      "notes": "Comprado no after-hours"

    ...
  ],
  "skipped": [
    {"notion_id": "...", "reason": "Asset relation missing — orphan row"}
  ]

```

### Idempotency

Match key: `(external_source = 'notion', external_id = <notion_id>)`. On match → update fields. Never duplicate.

### Asset resolution

For each Notion lançamento, `Ativo` is a relation to a `DB IG Ativos` page. The importer:

1. Reads the Notion URL of the related asset.
2. Looks up the local Asset by `(external_source='notion', external_id=<that url>)`.
3. If not found → skips with a `reason: "asset external_id not found locally"` entry, so the user can audit.

Assumes Spec 07a's import has been run and Assets carry their Notion `external_id`.

### Run modes

```
python scripts/import_notion_lancamentos.py --dry-run     # preview, no DB writes
python scripts/import_notion_lancamentos.py --apply       # write to DB
python scripts/import_notion_lancamentos.py --from-json PATH  # custom JSON path
```

Exit codes:
- `0`: clean
- `2`: any unmapped `Tipo Transação`
- `3`: any orphaned lançamento (asset external_id not in local DB)
- `4`: any **hard validation error** (see below)

### Data quality validations

Notion data was hand-entered over years; the importer must surface inconsistencies rather than silently swallow them. The snapshot JSON gains two arrays alongside `lancamentos`:

- `errors[]` — **hard errors**, row is skipped, exit code `4`
- `warnings[]` — **soft issues**, row is imported but flagged for the user

Each entry is `{notion_id, code, message, raw}`.

**Hard errors (skip + exit 4):**

| Code | Triggered when |
|---|---|
| `MISSING_DATE` | `Data Transação` is null |
| `FUTURE_DATE` | `Data Transação` > today |
| `MISSING_ASSET` | `Ativo` relation empty |
| `UNKNOWN_TYPE` | `Tipo Transação` not in the mapping table |
| `ORPHAN_ASSET` | Asset's Notion URL not found in local Asset table (run 07a first?) |
| `DUPLICATE_NOTION_ID` | Two source rows share the same Notion page URL (shouldn't happen) |
| `COMPRA_VENDA_MISSING_VALUE` | `COMPRA`/`VENDA`/`SUBSCRICAO`/`RESGATE_TOTAL` row that has neither `(Qtde + Preço Unit.)` nor `AR Valor` populated |
| `COMECOTAS_MISSING_TAX` | `ComeCota` row with `Taxas` null AND `AR Valor` null |
| `BONIFICACAO_MISSING_QTY` | `Bonificação` row with `Qtde` null/0 |

**Warnings (import + flag):**

| Code | Triggered when |
|---|---|
| `UNIT_PRICE_ZERO` | `Preço Unit. = 0` on a Cotado COMPRA/VENDA (likely typo or "free transfer" — surface for review) |
| `BONIFICACAO_HAS_PRICE` | `Bonificação` with non-zero `Preço Unit.` (rare — "valor de incorporação" cases — gross_amount uses qty × price as override) |
| `FX_RATE_MISSING_FOR_USD` | USD asset's lançamento has null `Cambio do Dia`; importer falls back to `1.0` but flags for later PTAX backfill |
| `FX_RATE_PRESENT_FOR_BRL` | BRL asset has non-1.0 `Cambio do Dia` (probably stale data; persisted as-is but flagged) |
| `BOTH_QTY_AND_AR_VALOR` | Row has both `(Qtde + Preço Unit.)` AND `AR Valor` set; importer prefers `(Qtde + Preço Unit.)` and ignores `AR Valor`, flagging the discrepancy |
| `NEGATIVE_RUNNING_QTY` | Cumulative position goes negative for an asset without a prior `RESGATE_TOTAL` reset; surfaces a likely missing/out-of-order COMPRA |
| `LARGE_FEE_RATIO` | `Taxas` > 5% of `Preço Unit. × Qtde` — likely a typo |
| `DATE_BEFORE_ASSET_CREATED` | `Data Transação` < the related asset's `created_at` in Notion (possible but worth flagging) |

**Implementation:**

- `errors[]` and `warnings[]` are populated during the **fetch step** (this is where I — Claude / orchestrator — have visibility) AND re-validated during `--apply` against local invariants (`ORPHAN_ASSET`, `NEGATIVE_RUNNING_QTY`).
- Both arrays land in the JSON for human inspection.
- `--dry-run` prints a summary table grouped by code with counts.
- `--apply` requires `errors.length == 0` to proceed (or pass `--force` to import despite errors — for the rare "I know better" case).
