# Spec 57 — Extract import: ticker normalization + auto-priced skip + duplicate disambiguator

## Status: ✅ Done (2026-06-07)

3 testes novos verdes em `tests/test_bulk_extract.py` + 12 em
`tests/test_extraction_disambiguate.py` (42 passes nas suites de
extraction/snapshot). Fix aplicado em
`src/numis_geek/services/extraction.py` e
`src/numis_geek/services/extraction_templates/__init__.py` (BROKER_POSITION
prompt v2 → v3). Sem migration, sem mudança de contrato HTTP —
comportamento puramente interno.

## Why

Bug encontrado tentando importar extrato da Avenue no fechamento de
mai/2026: **nenhum** dos ~15 ativos casou, apesar dos tickers (AAPL,
MSFT, VOO, etc.) baterem exatamente com o cadastro.

### Causa raiz 1 — ticker match case-sensitive sem strip

`_resolve_asset_by_ticker_or_name` (extraction.py:593-645, pré-fix)
tinha 3 passos: ticker exato → name lowercase → substring. O passo 1
era:

```python
Asset.ticker == candidate  # case-sensitive, sem strip
```

LLM emite frequentemente `"petr4"`, `" PETR4"` ou `"AAPL "`. Match
falha. Cai pro passo 2 (match por **nome**, não ticker), que exige
que `Asset.name.lower()` bata literalmente — `Asset.name="Apple Inc."`
vs extrato `"AAPL"` → falha. Cai pro passo 3 substring, que tem guard
`len<4` (`"AAPL"` passa, `"VOO"` não) e exige match único. Resultado:
maioria dos ativos cai como órfão.

### Causa raiz 2 — extrato sobrescreve preço de ativo auto-priced

`_apply_bulk_to_snapshot` carregava **todas** as pendencies abertas
sem olhar `action_type`:

```python
open_pendencies = db.query(SnapshotPendency).filter(
    SnapshotPendency.snapshot_id == snap.id,
    SnapshotPendency.resolved_at.is_(None),
).all()
```

Ativos com `price_source ∈ {BRAPI, FINNHUB, COINBASE, TESOURO}` cuja
API ficou stale geram pendency `action_type=RETRY_API` (snapshot.py
`detect_pendencies`). O fix correto pra esses ativos é re-tentar a
API, **não** subir um PDF — o provider é a fonte da verdade do preço.
Mas o bulk apply resolvia a pendency normalmente, sobrescrevendo o
preço da API.

Pior: ativos auto-priced **frescos** (sem pendency) também caíam no
bucket `matched_no_pendency`, gerando ruído visual no painel de review
("matched mas sem pendência" — confunde o usuário).

### Causa raiz 3 — LLM emite `ticker_raw` duplicado pra ativos distintos

Extrato real da Avenue (mai/2026): das 62 linhas extraídas, **10 vieram
com `ticker_raw="United States of America"`** — uma por T-Bill/T-Note
americano. O LLM identificou corretamente cada vencimento no `notes`
(`"Tesouro US - Vencimento: 16/08/2034"`, …) mas só uma linha veio com
vencimento no `ticker_raw` propriamente dito; as outras nove só com
"United States of America". Mesmo padrão pra debêntures (Marcopolo) e
bonds corporativos (JPMorgan Chase).

Prompt v2 cobria CDB/LCI/LCA mas não Treasury US, bonds corporativos
ou debêntures explicitamente, e não tinha regra geral de
"`ticker_raw` precisa ser único".

## Decisões locked

| # | Pergunta | Resposta |
|---|---|---|
| 1 | Normalização do ticker | `candidate.strip().lower()` no Python + `func.lower(func.trim(Asset.ticker))` no SQL. Match dos dois lados. Escopado a `is_active=True`. |
| 2 | Auto-priced no bulk | `_apply_bulk_to_snapshot` exclui `action_type=RETRY_API` da query de open pendencies. Dentro do loop, ativo auto-priced que aparece no extrato sem `manual_pendency_id` é pulado silenciosamente (não vai pra `matched_no_pendency` nem `orphan`). |
| 3 | Auto-priced no path singular | `_apply_broker_position` (path Spec 38 sem snapshot) também pula auto-priced — adiciona ao `errors[]` com mensagem explícita ("preço gerenciado por FINNHUB — extrato ignorado"). |
| 4 | Override manual | Se o user mapeou manualmente a pendency via `manual_mappings` (Spec 49), respeita — assume que o user sabe o que faz. |
| 5 | RETRY_API pendency em `pendency_not_in_extract` | **Não aparece**. Filtrada na query inicial, não vaza pra nenhum bucket. |
| 6 | Prevenção de `ticker_raw` duplicado | **Two-pronged**: prompt v3 estabelece "Regra de Ouro — duas posições NUNCA podem ter o mesmo `ticker_raw`" + cobre Treasury/bonds/debêntures explicitamente. Como rede de proteção, `_disambiguate_duplicate_tickers` roda pós-validação e anexa vencimento extraído de `notes` quando colisões ocorrerem. |
| 7 | Disambiguador sem data no `notes` | **Não renomeia**. Deixa o `ticker_raw` original — melhor uma órfã honesta do que um rename inventado. |

## In Scope

### Backend — `src/numis_geek/services/extraction.py`

**1. `_resolve_asset_by_ticker_or_name`**

```python
needle = candidate.strip().lower()
hit = (
    db.query(Asset)
    .filter(
        Asset.workspace_id == workspace_id,
        Asset.is_active == True,
        func.lower(func.trim(Asset.ticker)) == needle,
    )
    .first()
)
```

Passos 2 (name match) e 3 (substring) inalterados — só o passo 1
ganha normalização.

**2. `_apply_broker_position`** (singular path — Spec 38)

- Substitui query inline por `_resolve_asset_by_ticker_or_name` (ganha
  normalização de graça).
- Antes do `asset.current_price = ...`, checa
  `asset.price_source in AUTOMATED_SOURCES`; se sim, incrementa
  `skipped` e loga em `errors[]`.

**3. `_apply_bulk_to_snapshot`** (bulk path — Spec 48)

- Query de `open_pendencies` filtra `action_type != RETRY_API`.
- Dentro do loop, após resolver o asset: se `price_source ∈ AUTOMATED_SOURCES`
  e `not manual_pendency_id`, `continue` silencioso.

### Backend — `src/numis_geek/services/extraction_templates/__init__.py`

**`BROKER_POSITION` template v2 → v3**

- "Regra de Ouro" no topo: duas posições NUNCA podem ter o mesmo
  `ticker_raw`; se duplicaria, adicionar vencimento/cupom/ISIN.
- Regras explícitas pra:
  - US Treasury (T-Bills/T-Notes/T-Bonds): `US Treasury YYYY-MM-DD`
  - Bonds corporativos US: `Emissor YYYY-MM-DD`
  - Debêntures BR: `Emissor YYYY-MM-DD`
- Mantém regras existentes pra ações, ETFs, fundos, Tesouro BR,
  CDB/LCI/LCA.

### Backend — `src/numis_geek/services/extraction.py`

**`_disambiguate_duplicate_tickers(validated)`**

Roda imediatamente após `template.output_model.model_validate(...)`,
antes de persistir em `job.extracted_json`. Detecta `ticker_raw`
duplicados, extrai vencimento do `notes` via 3 regexes (ISO
`YYYY-MM-DD`, BR `DD/MM/YYYY`, parcial `MM/YYYY`) e anexa ao
`ticker_raw`. Colisão com `ticker_raw` já único existente → adiciona
sufixo `#2`, `#3`. Sem `notes` ou sem data extraível → noop (deixa
duplicado, vai pra orphan). Seguro pra output models sem `positions`.

## Out of Scope

- **Frontend warning** quando ativo auto-priced é encontrado no
  extrato. Comportamento atual é silencioso. Se virar dor, abrir spec
  separada pra logar/mostrar no painel.
- **Reverter sobrescritas antigas**: alguns ativos auto-priced em
  fechamentos passados podem ter sido tocados pelo bulk antes do fix.
  Não há backfill — `PortfolioSnapshotItem` é frozen (Spec 52) e o
  preço atual do `Asset` será sobrescrito no próximo refresh da API.
- **Fuzzy ticker match** (Levenshtein, prefix variants). O fix atual
  cobre as variações encontradas em produção. Se aparecer novo caso,
  adicionar regra de normalização específica.

## Risks / Mitigations

| Risco | Mitigação |
|---|---|
| Match case-insensitive pode colidir com tickers que diferem só por case (`mglu3` vs `MGLU3` cadastrados como ativos distintos) | Improvável — convenção é uppercase. Se acontecer, `.first()` pega um deles; auditoria via `bulk_detail.applied` vai mostrar o asset_id real. |
| Ativo auto-priced legítimamente sem cotação da API (provider quebrou, asset migrou de fonte) | User pode editar `price_source=MANUAL` no admin do ativo, ou usar override manual via `manual_mappings`. |
| Pendency `RETRY_API` ficar permanentemente aberta se a API nunca recuperar | Out of scope desta spec — é problema do refresh job. |
