# Spec 49 — Bulk Upload UX Rework

## Status: 📝 Implementing

## Why

Spec 48 entregou bulk extract num clique único: dropar → LLM → modal de
review. Na prática o fluxo é **opaco**:
- Após o upload, nada visível na UI. O user fica olhando "Extraindo…" no
  modal sem entender o que está rolando.
- Se a extração falha (JSON inválido, validação pydantic, timeout), o
  anexo desaparece sem rastro — perdeu o arquivo, tem que subir de novo.
- Não dá pra subir vários arquivos pra revisar depois. Cada um vira um
  ciclo upload→modal→aplicar→fechar antes do próximo.
- Sem preview da imagem antes de processar.

## Decisões locked

| # | Pergunta | Resposta |
|---|---|---|
| 1 | Persistência dos anexos | **DB** — linkados ao snapshot. Sobrevivem reload, servem como prova de auditoria do fechamento. |
| 2 | Quando rodar a extração | Botão **Extrair** explícito por anexo. Modal de review abre na hora (sync). |
| 3 | Status visual | **Resumo curto inline** ("Casadas 12 · Órfãs 3") + detalhe completo no modal de review ao clicar Revisar. |
| 4 | Multi-arquivo | **Sequencial**: 1 extração por vez. Enquanto uma roda, demais ficam "Aguardando" (botão Extrair desabilitado). |

## In Scope

### 1. Backend

**1.1 `GET /snapshots/{snapshot_id}/attachments`** → `list[AttachmentOut]`

Lista anexos com `source_type=SNAPSHOT, source_id=snapshot_id` e
`is_active=true`. Reusa a query existente em `routes/attachments.py`.

**1.2 `GET /snapshots/{snapshot_id}/extractions`** → `list[BulkExtractionJobOut]`

Lista `ExtractionJob` onde `snapshot_id=snapshot_id` AND `pendency_id IS NULL`
(os jobs bulk). Shape leve — só id, attachment_id, status, contagens
preview (sem o JSON completo até o user clicar Revisar):

```python
class BulkExtractionJobOut(BaseModel):
    id: str
    attachment_id: str
    status: str               # PENDING/RUNNING/EXTRACTED/CONFIRMED/REJECTED/FAILED
    positions_count: int      # len(extracted_json.positions) — 0 se ainda não rodou
    error_message: str | None
    created_at: str
    completed_at: str | None
    confirmed_at: str | None
```

**1.3 `POST /snapshots/{snapshot_id}/bulk-extract` (já existe)**

Sem mudança no shape do endpoint. Mantém comportamento sync de criar
+ rodar.

**1.4 Upload de anexo escopado**

Reusa o `POST /attachments` existente com `source_type=snapshot` e
`source_id=snapshot.id`. Sem endpoint novo.

### 2. Frontend

**2.1 `<BulkAttachmentManager />`** — substitui `BulkUploadZone` no
`PendencyPanel`.

Composição interna:
- Header "Anexos do fechamento"
- **Dropzone amarela** (drag-and-drop + click + paste) — igual hoje,
  mas só faz upload. Não dispara extração.
- **Lista** de anexos abaixo, carregada via
  `GET /snapshots/{id}/attachments` + `GET /snapshots/{id}/extractions`.
- Cada linha (`<AttachmentRow />`):
  ```
  [icon|thumb] filename.png · 1.2MB                      <status badge>
                                  [Preview] [Extrair / Revisar] [🗑]
  ```

**2.2 Status badges (mapeados do job)**

| Job status | Badge | Botão de ação |
|---|---|---|
| _Sem job_ | (vazio) | **Extrair** (azul) |
| `PENDING`/`RUNNING` | `⏳ Extraindo…` (animado) | desabilitado |
| `EXTRACTED` | `✨ {N} posições` | **Revisar** (verde) + **Re-extrair** (cinza pequeno) |
| `CONFIRMED` | `✓ Aplicado` (verde) | **Re-extrair** (apenas, com aviso) |
| `FAILED` | `⚠️ Falhou` + tooltip do erro | **Tentar novamente** |
| `REJECTED` | `⊘ Descartado` | **Extrair** novamente |

> **Por que mostrar `positions_count` em vez de `casadas`?** Backend
> não pré-calcula matching (caro: precisaria carregar todas as
> pendencies abertas pra cada job). Frontend mostra contagem total da
> extração; o match preview vive no modal de review (já implementado
> em `<BulkExtractReviewModal />`).

**2.3 Fila sequencial**

State `extractingAttachmentId: string | null` no `BulkAttachmentManager`.
Enquanto não-nulo, todos os botões "Extrair" das outras linhas viram
"Aguardando" (texto cinza, disabled). Garante:
- Sem rajadas paralelas no Anthropic
- User entende o que está rodando
- Custos previsíveis

**2.4 Preview**

V1: clique em **Preview** abre `GET /attachments/{id}/download` em nova
aba. Browser renderiza PNG/JPG/PDF inline. Pra spec 50 futuro: modal
inline com next/prev pra navegar entre anexos.

**2.5 Remover**

`DELETE /attachments/{id}` (já existe). Soft-delete. Se houver
ExtractionJob CONFIRMED apontando pra esse attachment, **não deleta** —
mostra aviso "Esse anexo está vinculado a N pendências resolvidas. Re-abra
o snapshot pra remover.". V1 só bloqueia; UX de reabrir fica pra depois.

**2.6 Revisar**

Clique no botão Revisar (status=EXTRACTED) abre `<BulkExtractReviewModal>`
com o job existente — sem nova chamada de LLM. O modal já recebe `job`
como prop, então é só armazenar a referência no state do
`BulkAttachmentManager`.

### 3. Mudanças em componentes existentes

- **`<PendencyPanel>`**: troca `<BulkUploadZone>` por

- **`<BulkUploadZone>`**: deletado.
- **`<BulkExtractReviewModal>`**: sem mudança.

## Out of Scope

- **Modal de preview rico** (inline image viewer, zoom, anotações).
  V1 abre em nova aba. → Spec 50 (futuro).
- **Drag-reorder** dos anexos. Ordem = uploaded_at desc.
- **Extração paralela** (fila > 1). Sequencial só.
- **Edit inline do JSON extraído** antes de aplicar. Spec 38 V1 já
  tinha esse plano e foi cortado; mantém cortado.

## Acceptance Criteria

1. Refresh da página `/snapshots/{ym}` mostra todos os anexos
   subidos previamente + seus status de extração.
2. Subir 3 prints sem clicar Extrair = 3 linhas com badge vazio +
   botão Extrair em cada uma.
3. Clicar Extrair no #1 enquanto #2 e #3 mostram "Aguardando" (botão
   disabled).
4. #1 termina, badge vira `✨ {N} posições` + botões Revisar e
   Re-extrair. Botão Extrair do #2 reabilita.
5. Revisar abre o modal com o job já carregado; aplicar fecha
   pendências e atualiza counters do painel.
6. Remover anexo CONFIRMED bloqueado com mensagem clara.
7. `pytest -q` + `npm run test:run` + `npm run build` verdes.
