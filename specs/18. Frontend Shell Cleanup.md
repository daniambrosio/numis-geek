### 4. `MovementComposer` — apenas 6 tipos normais (revisado 2026-05-23)

**Arquivo:** `frontend/src/components/MovementComposer.tsx` (substitui
`LancamentoModal.tsx`).

> **Mudança de escopo durante a implementação:** o protótipo
> (`prototypes/index.html:4242-4387`) tem **apenas os 6 tipos normais**
> no MovementComposer. Os 6 tipos de opção (SELL_OPEN/BUY_TO_OPEN/
> SELL_TO_CLOSE/BUY_TO_CLOSE/EXERCISED/EXPIRED) **não aparecem aqui** —
> opções têm um fluxo dedicado:
> - **Criar OPTION asset + primeiro movement (SELL_OPEN/BUY_TO_OPEN)** →
>   `OptionModal` aberto via botão "+ Opção" no `AssetDetail` do
>   underlying.
> - **EXERCISED / EXPIRED** → ações no `OpenOptionsCard` (já existem).
> - **BUY_TO_CLOSE / SELL_TO_CLOSE** → ficam pra spec dedicada de
>   lifecycle de opção (encerrar posição cedo).
>
> Razão: o usuário (review 2026-05-23) confirmou que misturar 12 tipos
> num composer só polui. Tipos de opção são conceitualmente diferentes
> (qty representa contratos × size, prêmio ≠ preço unitário no sentido
> tradicional), e EXERCISED/EXPIRED são *ações* sobre uma posição
> aberta, não tipos que o usuário escolhe de cara.

**Layout (espelho do protótipo):**

```
┌─ Tipo ─ (grid 3×2, cards com label + hint) ───────────┐
│  [Compra]      [Venda]        [Bonificação]            │
│  compra adic.  venda reduz    ações grátis             │
│                                                         │
│  [Subscrição]  [Come-cotas]   [Resgate Total]          │
│  exercício     imposto sem.   vencimento ou liq.       │
└────────────────────────────────────────────────────────┘
[ Data do evento ]   [ Ativo ]
[ Quantidade ]       [ Preço unitário · CCY ]
[ Taxa · CCY · opcional ]
[ Imposto retido · CCY ]            ← só COME_COTAS
[ ⚠ ativo zerado ]                  ← só se asset inactive
[ Nº Nota de Corretagem · opcional ]← BUY/SELL/SUBSCR/FULL_REDEMPTION

✨ PRÉ-VISUALIZAÇÃO
   Net               +R$ X,XX
   Posição           Δ +100 PETR4

NOTAS · opcional · ⌘V cola imagem
[ textarea ]

⌘↵ salvar · esc fechar       [ Cancelar ] [ ✓ Salvar ]
```

**Form reshape** por tipo:

| Bloco | Aparece em |
|---|---|
| Quantidade + Preço unitário + Taxa | BUY, SELL, SUBSCRIPTION, FULL_REDEMPTION |
| Quantidade só (sem preço/taxa) | BONUS |
| Imposto retido (no qty/price) | COME_COTAS |
| Valor único (substitui qty×price) | BUY/SELL/SUBSCR/FULL_REDEMPTION quando asset é não-cotado (FIXED_INCOME, FGTS, PRIVATE_PENSION, CASH) |
| Nº Nota de Corretagem | BUY, SELL, FULL_REDEMPTION, SUBSCRIPTION |
| Pré-visualização (Net + Posição) | sempre que asset selecionado |

**Asset filter:** lista **todos** os ativos, incluindo OPTION
(`prototypes/index.html:4312-4316`). O usuário pode lançar Compra/Venda
sobre uma opção da mesma forma que sobre uma ação — o que distingue é a
asset_class (com `option_type` CALL/PUT como metadado do asset).

> Decisão (2026-05-23, revisada): opção **É** um asset normal pra fins do
> composer. Os 6 movement types option-specific do schema
> (SELL_OPEN/BUY_TO_OPEN/etc.) ficam pra fluxos especializados — o
> backend deve interpretar/mapear quando necessário. O **botão "+ Opção"
> no AssetDetail** (já existente) continua como atalho pra criar
> simultaneamente o asset OPTION + primeiro movement com parser BR.

**Submit:** `POST /asset_movements` com os 6 types normais.

### 6. Notion sync — option lifecycle types skipped (revisado 2026-05-23)

Durante o teste manual, o bulk sync de 2 movimentos `SELL_OPEN` (ITUBR364
e ITUBF475) deu HTTP 500 — `KeyError` em `_MOVEMENT_TYPE_PT` que só tinha
os 6 tipos clássicos. Fix em camadas:

1. **Adicionado** `NotionSyncStatus.SKIPPED` (enum).
2. **Migration** `e8f9a0b1c2d3_notion_sync_skipped` aplicada nas 5 tabelas
   que carregam `notion_sync_status` (asset, asset_movement,
   portfolio_snapshot, portfolio_snapshot_item, corporate_action).
3. **`push_asset_movement`** detecta option lifecycle types
   (SELL_OPEN/BUY_TO_OPEN/SELL_TO_CLOSE/BUY_TO_CLOSE/EXERCISED/EXPIRED) e
   marca como SKIPPED antes de tocar no Notion API.
4. `list_pending` continua filtrando só `PENDING|ERROR` — SKIPPED não
   entra no contador "Sync Notion (N)".
5. Documentado em `docs/options-rationale.md` §3.3.

Backup do DB tomado em `numis_geek.db.bak-before-18-skipped` antes da
migration.
