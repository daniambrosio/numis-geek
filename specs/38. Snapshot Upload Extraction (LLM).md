## Status: ✅ Done (V1) — 2026-05-26

## Alignment decisions (resolved autonomously 2026-05-26)

1. **Sync vs async** — **sync** em V1 (chamada inline no POST). Files
   típicos (1 screenshot) cabem em < 30 s; APScheduler async fica pra
   V2 quando upload em batch virar caso comum.
2. **Modelo** — `claude-sonnet-4-5` como default (ver `DEFAULT_MODEL` em
   `integrations/llm.py`). Opus fica disponível via override.
3. **Confidence thresholds** — > 0.9 verde, 0.7-0.9 âmbar, < 0.7 vermelho.
   Sem auto-aviso "revisar manualmente" — sempre exige review.
4. **Source hints V1** — todos os 5 expostos no dropdown, mas só
   `SCREENSHOT_PRICE` e `BROKER_POSITION` têm prompt produção; os
   outros 3 (`BROKER_INCOME`, `B3_TRADE_NOTE`, `FGTS_BALANCE`) têm
   schema definido + prompt placeholder com TODO marcado. `GENERIC`
   é alias para `BROKER_POSITION` por enquanto.
5. **Ticker normalization** — **skip + report** (não auto-cria Asset
   stub). Tickers desconhecidos viram entrada no `errors[]` do apply
   result.
6. **PII** — V1 manda CPF/conta como veio (Anthropic não treina em API
   content por default). Risco documentado em
   `docs/extraction-prompts.md` §"Known limitations".
7. **Prompts** — hardcoded em `extraction_templates/__init__.py` com
   `prompt_version` versionado por template. Mover pro DB se virar
   pedido recorrente.
8. **Custo médio** — só track (`cost_usd` por job), sem alerta de
   billing em V1.
9. **Audit log** — sim, cada `confirm`/`reject` vira entry com
   `details = { applied, skipped, errors, pendency_id }`.
10. **Reutilizar attachment** — sim, vários `ExtractionJob` podem
    apontar pra mesmo `attachment_id` (o front cria um novo job na
    tentativa seguinte).

## Overview

Spec 35 deixou o botão **"Upload extrato"** no painel de pendências
mas a ação não faz nada além de criar `Attachment`. Esta spec entrega
o **extractor LLM** que transforma o arquivo em dados estruturados,
mostra preview pra revisão e (após confirmação) aplica os dados no
modelo (Asset price, Distribution, AssetMovement) — resolvendo a
pendência do snapshot.

Esta é a peça mais cara da fase de fechamentos: mexe em background
job, prompt engineering, schema de extração, UI de review/edit e
fluxo de aplicação dos dados extraídos.

**Filosofia**: extração é sempre **review-then-commit**. Nada é
gravado automaticamente — o usuário sempre vê o que o LLM extraiu,
edita se necessário, e confirma. Confidence score informa, não
decide.

## In Scope

### 1. Schema — `ExtractionJob`

```python
# src/numis_geek/models/extraction_job.py

class ExtractionStatus(str, enum.Enum):
    PENDING    = "PENDING"     # criado, aguardando worker
    RUNNING    = "RUNNING"     # LLM em chamada
    EXTRACTED  = "EXTRACTED"   # JSON pronto, aguardando review do usuário
    CONFIRMED  = "CONFIRMED"   # usuário aplicou; dados gravados
    REJECTED   = "REJECTED"    # usuário descartou (extração ruim)
    FAILED     = "FAILED"      # LLM erro / parse error / timeout

class ExtractionSourceHint(str, enum.Enum):
    BROKER_POSITION       = "BROKER_POSITION"        # extrato de posições (qty + preço)
    BROKER_INCOME         = "BROKER_INCOME"          # extrato de rendimentos/proventos
    B3_TRADE_NOTE         = "B3_TRADE_NOTE"          # nota de corretagem (trades)
    FGTS_BALANCE          = "FGTS_BALANCE"           # saldo FGTS
    SCREENSHOT_PRICE      = "SCREENSHOT_PRICE"       # screenshot do app do broker
    GENERIC               = "GENERIC"                # tenta detectar sozinho

class ExtractionJob(Base):
    __tablename__ = "extraction_job"
    id: Mapped[str] = mapped_column(String(36), primary_key=True, ...)
    workspace_id: Mapped[str] = mapped_column(ForeignKey("workspace.id"), nullable=False)

    # What triggered this job (one of):
    snapshot_id: Mapped[str | None]  = mapped_column(ForeignKey("portfolio_snapshot.id"), nullable=True)
    pendency_id: Mapped[str | None]  = mapped_column(ForeignKey("snapshot_pendency.id"), nullable=True)
    asset_id:    Mapped[str | None]  = mapped_column(ForeignKey("asset.id"), nullable=True)

    # The uploaded file (FK to existing Attachment from Spec 19)
    attachment_id: Mapped[str] = mapped_column(ForeignKey("attachment.id"), nullable=False)

    source_hint: Mapped[ExtractionSourceHint] = mapped_column(
        Enum(ExtractionSourceHint), nullable=False, default=ExtractionSourceHint.GENERIC,
    )
    status: Mapped[ExtractionStatus] = mapped_column(
        Enum(ExtractionStatus), nullable=False, default=ExtractionStatus.PENDING,
    )

    # LLM metadata
    model:           Mapped[str | None]      = mapped_column(String(100), nullable=True)
    prompt_version:  Mapped[str | None]      = mapped_column(String(20), nullable=True)
    input_tokens:    Mapped[int | None]      = mapped_column(Integer, nullable=True)
    output_tokens:   Mapped[int | None]      = mapped_column(Integer, nullable=True)
    cost_usd:        Mapped[Decimal | None]  = mapped_column(Numeric(10, 4), nullable=True)

    # Extracted payload (JSON shape varies by source_hint — see §3)
    extracted_json:  Mapped[dict | None]     = mapped_column(JSON, nullable=True)
    confidence:      Mapped[Decimal | None]  = mapped_column(Numeric(3, 2), nullable=True)  # 0.00..1.00
    detected_hint:   Mapped[ExtractionSourceHint | None] = mapped_column(
        Enum(ExtractionSourceHint), nullable=True,
    )  # what the LLM thought it was (may differ from source_hint when GENERIC)

    # Error tracking
    error_message:   Mapped[str | None]      = mapped_column(Text, nullable=True)
    retry_count:     Mapped[int]             = mapped_column(Integer, default=0, nullable=False)

    # Lifecycle
    created_at:    Mapped[datetime] = mapped_column(DateTime, ...)
    started_at:    Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
    completed_at:  Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
    confirmed_at:  Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
    confirmed_by:  Mapped[str | None]      = mapped_column(String(36), nullable=True)
    user_edits:    Mapped[dict | None]     = mapped_column(JSON, nullable=True)
        # final payload that was applied (== extracted_json XOR user edits)

    __table_args__ = (
        Index("ix_extraction_workspace_status", "workspace_id", "status"),
        Index("ix_extraction_pendency", "pendency_id"),
    )
```

**Migration `<hash>_extraction_job.py`** — cria tabela + enums.

### 3. Templates de extração por `source_hint`

**3.1 `BROKER_POSITION` — extrato de posição**

Schema esperado:

```json
{
  "as_of_date": "2026-04-30",
  "broker_name": "XP Investimentos",
  "positions": [
    {
      "ticker_raw": "PETR4",
      "ticker_normalized": "PETR4",
      "quantity": 200,
      "unit_price": 38.45,
      "currency": "BRL",
      "market_value": 7690.00,
      "confidence": 0.95,
      "notes": null
    }
  ],
  "summary_total_brl": 142350.00,
  "summary_total_usd": null
}
```

Prompt (resumo): "Você é um extrator de extratos de corretora.
Extraia cada posição como linha JSON. Use ticker B3 canônico
(PETR4, ITUB4, BBSE3...). Para FIIs use ticker com sufixo 11.
Para ativos americanos use ticker NASDAQ/NYSE puro (AAPL, MSFT).
Se não conseguir identificar o ticker, deixe `ticker_normalized`
como null e ponha confidence baixa..."

**3.2 `BROKER_INCOME` — extrato de proventos**

```json
{
  "as_of_date": "2026-04-30",
  "broker_name": "Avenue Securities",
  "events": [
    {
      "event_date": "2026-04-15",
      "ticker_raw": "AAPL",
      "ticker_normalized": "AAPL",
      "type": "DIVIDEND",
      "gross_amount": 24.50,
```

**3.6 `GENERIC` — fallback**

Prompt pede ao LLM pra **primeiro classificar** o documento
em uma das 5 categorias (ou "UNKNOWN"), e em seguida extrair no
schema apropriado. Retorna `detected_hint` populado.

### 4. Worker — extração assíncrona

**4.1 Service `services/extraction.py`**

```python
def create_extraction_job(
    db, *, workspace_id, attachment_id, source_hint,
    pendency_id=None, snapshot_id=None, asset_id=None,
    user_id=None,
) -> ExtractionJob: ...

def run_extraction(db, job_id: str) -> ExtractionJob:
    """
    - Carrega job + attachment.
    - Lê o arquivo (PDF/XLSX/PNG/JPG/CSV) e converte para input do LLM:
        - PDF: pdf_to_images() (uma página → 1 imagem) ou pdf_to_text() se
          é PDF textual puro.
        - XLSX: openpyxl → CSV em memória.
        - PNG/JPG: passa direto como image input.
        - CSV: passa como texto.
    - Monta prompt = template[source_hint] + content + JSON schema.
    - Chama Anthropic API (Sonnet 4.x).
    - Parse + valida com pydantic schema.
    - Salva extracted_json, confidence, tokens, cost.
    - Status → EXTRACTED ou FAILED.
    """

def confirm_extraction(
    db, *, job_id, user_id, edited_payload=None,
) -> ExtractionApplyResult:
    """
    - Pega job em EXTRACTED.
    - Aplica o payload (edited_payload se passado, senão extracted_json):
        - BROKER_POSITION → cria/atualiza Asset.current_price para cada ticker
          (resolve ticker → Asset; se não encontrar, gera Asset stub ou
          flag de erro com link "Criar ativo").
        - BROKER_INCOME → cria Distribution rows.
        - B3_TRADE_NOTE → cria AssetMovement rows.
        - FGTS_BALANCE → atualiza Asset (klass=FGTS) current_price para
          o saldo informado.
        - SCREENSHOT_PRICE → atualiza Asset.current_price.
    - Se job.pendency_id existe: marca SnapshotPendency como resolvida
      (chama services/snapshot.resolve_pendency com note="extraído via LLM").
    - Status → CONFIRMED, confirmed_at, confirmed_by.
    - Retorna summary: { applied_count, skipped_count, errors }.
    """

def reject_extraction(db, *, job_id, user_id, reason) -> None:
    """Status → REJECTED. Pendência permanece aberta."""
```

**4.2 Background runner**

- Reusar APScheduler da Spec 35 (Fase B) — adicionar interval job
  que polla `SELECT * FROM extraction_job WHERE status = 'PENDING'`
  e processa em série (concorrência 1 pra começar; LLM é
  rate-limitado).
- Alternativa: rodar inline no endpoint que cria o job (sync) se
  o arquivo é pequeno (< 1MB e < 5 páginas) — UX melhor, sem
  precisar polling. Trade-off na §plan-mode.

### 5. API endpoints

```
POST   /extractions                              criar job
       body: { attachment_id, source_hint, pendency_id?, snapshot_id?, asset_id? }
       returns: ExtractionJobOut (status=PENDING ou RUNNING/EXTRACTED se sync)
```

Autorização:
- Workspace-scoped (sysadmin atravessa workspaces).
- Quem cria job é quem pode confirmar/rejeitar (ou admin do

## Out of Scope

- **Auto-confirm** quando confidence > 0.95. Sempre exigir review
  no V1. Reduz risco de regredir patrimônio por extração ruim.
- **Multi-asset detection** em screenshot (1 imagem com 5 tickers).
  V1 trata 1 ticker por screenshot.
- **Reconciliação de extração com snapshot anterior** (ex.: avisar
  "qty era 200, virou 150, era venda?"). Útil mas complexo —
  vira spec própria.
- **Suporte a fontes brasileiras específicas** (CEI/B3 nota
  oficial com formato XML) — tratado como BROKER_POSITION genérico
  por enquanto.
- **OCR pré-processamento** para PDFs digitalizados ruins. Claude
  Sonnet já faz vision; suficiente pra V1.
- **PII redaction** antes de enviar pra Anthropic. Anthropic não
  treina em API content por default; aceitar como risco no V1 e
  documentar.
- **Rate limit por workspace** — feature de billing futura.
- **Webhook callback** quando extração termina (ao invés de polling).
  Polling V1; webhook se UX provar problemático.

## Acceptance Criteria

1. Migration aplica em DB existente sem perda.
2. `POST /extractions` cria job com status=PENDING e retorna ID.
3. Worker processa job: chama LLM, salva JSON, status=EXTRACTED.
4. `GET /extractions/{id}` retorna payload completo quando EXTRACTED.
5. Para cada um dos 5 source hints, há um template que extrai
   corretamente um sample real (golden file).
6. `POST /extractions/{id}/confirm` aplica os dados:
   - BROKER_POSITION → Asset.current_price atualizado, ≥1 ativo
   - BROKER_INCOME → Distribution rows criadas, ≥1 evento
   - B3_TRADE_NOTE → AssetMovement rows criadas, ≥1 trade
   - FGTS_BALANCE → Asset (klass=FGTS) current_price atualizado
   - SCREENSHOT_PRICE → Asset.current_price atualizado
7. Confirm resolve a SnapshotPendency vinculada (se houver):
   `resolved_at` populado, snapshot recalcula totais.

## Pre-implementation interview (Planner agent)

Decisões pra alinhar com o user antes de codar:

1. **Sync vs async em V1?**
   - Sync (chama LLM no endpoint, bloqueia request 10-30s): UX
     mais simples (sem polling), mas timeouts são risco.
   - Async + polling (recomendado): mais robusto, exige polling
     no frontend.
   - Híbrido: sync se arquivo < 1MB, async caso contrário.

2. **Modelo LLM**: Sonnet 4.5 (mais barato, suficiente pra
   maioria) vs Opus (mais caro mas mais preciso em casos
   ambíguos)? Configurável por hint?

3. **Confidence threshold**: qual cor pra cada tier?
   Sugestão: >0.9 verde, 0.7-0.9 âmbar, <0.7 vermelho.
   Algum corte ativa um aviso "revisar manualmente"?

4. **Source hints V1**: confirmar os 5 (BROKER_POSITION,
   BROKER_INCOME, B3_TRADE_NOTE, FGTS_BALANCE, SCREENSHOT_PRICE).
   Adicionar BANK_STATEMENT (extrato bancário CSV)?

5. **Ticker normalization**: o que fazer com ticker_raw que
   não bate? Opções:
   - Skip + report (V1 recomendado)
   - Auto-criar Asset stub (perigoso — pode duplicar)
   - Modal pra usuário fazer match manual

6. **PII**: aceitar enviar CPF/conta pra Anthropic sem redact
   no V1? Documentar e seguir, ou implementar redação básica
   (regex CPF/conta) antes de mandar?

7. **Storage de prompts**: hardcoded em código vs DB (editável
   por admin)? V1 hardcoded é mais simples mas exige redeploy
   pra ajustar.

8. **Custo médio aceitável por extração**: setar budget alerta
   se mensal > $X? V1 só track, sem alerta.

9. **Audit log**: cada confirm/reject vira entry?

10. **Reutilizar arquivo**: usuário fez upload mas rejeitou.
    Posso re-extrair com hint diferente sem novo upload?
    Sugestão: sim, novo job apontando pro mesmo `attachment_id`.
