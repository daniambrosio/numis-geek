# Spec 37 — Composer Attachments & Edit Mode

## Status: ✅ Done — 2026-05-26

## Alignment decisions (resolved autonomously 2026-05-26)

User estava dormindo durante a implementação; decisões abaixo seguiram as
sugestões da entrevista do rodapé (ver §"Pre-implementation interview"):

1. **Submit chain** — parcial com aviso inline. Registro persiste; falha
   parcial em anexos vira amber warning, usuário pode reabrir e re-tentar.
2. **Anexos persistidos no edit** — DELETE soft imediato (Spec 19) via
   `api.deleteAttachment`. Audit log da Spec 19 já cobre.
3. **Atalhos** — `⌘↵` e `Esc` mantidos sem mudança em modo edit.
4. **Confirmação de close com drafts não salvos** — NÃO implementado nesta
   primeira passada (drafts são in-memory e descartar é cheap). Reabrir
   spec se virar pedido recorrente.
5. **PUT payload** — full replace (já era o comportamento existente).
6. **Audit diff granularidade** — só os campos que mudaram (`utils/audit_diff.py`).
   Comparação por Decimal (não string) para evitar falso positivo entre
   `20.00000000` e `20.0`.
7. **Editar tipo de movimento** — o composer continua desabilitando o
   `type` picker em modo edit (estava assim antes). Sem aviso de
   recálculo porque o backend reprocessa net_amount em qualquer PUT.
8. **Persistidos vs drafts** — persistidos no topo (fundo cinza),
   drafts logo abaixo (fundo indigo + badge "novo").
9. **Audit granular pra add/remove anexo** — reusa endpoint Spec 19
   que já registra `attachment.uploaded` / `attachment.deactivated`.

## Escopo aplicado vs spec original

A spec original menciona 4 composers (Movement, Distribution, Transaction,
CardTx) + `ComposerProvider.edit()`. No app real só Movement e Distribution
existem; Transaction/CardTx ainda são placeholders no `NovoButton` (em
breve). `ComposerProvider` também não existe — cada page gerencia o seu
próprio estado de composer. Esses 2 sub-itens ficam **out-of-scope desta
spec** e serão reabertos quando os outros composers nascerem.

Detail panels já tinham `onEdit` props e botão Editar (Spec 17c/21) — esse
trabalho ficou só de wire dos `openEdit()` helpers que também carregam os
anexos persistidos do registro selecionado.

## Overview

Hoje o app real tem dois pontos cegos de UX que o protótipo já cobriu
e precisam virar código:

1. **Os 4 composers (Movement, Distribution, Transaction, CardTx)
   só permitem anotar texto** — sem upload de comprovante/extrato/
   nota de corretagem na criação. O usuário precisa salvar primeiro,
   abrir o detail panel, e só ali anexar via `<NotesAttachments>`.
   Friccional e contraintuitivo.
2. **Os detail panels (Lançamento, Provento) não têm botão Editar.**
   Para corrigir um campo errado o usuário precisa inativar e
   recriar.

Esta spec resolve os dois com um único componente reusável
(`<NotesAttachmentsField>`) consumido pelos composers, mais um modo
`editing` aplicável aos 4 composers, mais botão Editar no footer dos
detail panels.

**Nenhuma mudança de schema.** Reusa `Attachment` (Spec 19) e os
endpoints existentes de update de cada entidade.

## In Scope

### 1. `<NotesAttachmentsField>` — componente reusável

Em `frontend/src/components/NotesAttachmentsField.tsx`:

```typescript
interface Props {
  notes: string
  onNotesChange: (s: string) => void
  files: AttachmentDraft[]                 // pendentes (ainda não persistidos)
  onFilesChange: (xs: AttachmentDraft[]) => void
  placeholder?: string
}

interface AttachmentDraft {
  // Local-only until form save. Tipo igual ao Attachment do servidor
  // mas com File-like blob em vez de storage_key.
  id: string                     // local uuid
  file: File                     // browser File object
  name: string
  size: number
  mime_type: string
  kind: "image" | "pdf" | "csv" | "other"
  preview_url?: string           // object URL pra <img> em images
}
```

Comportamento:

- Textarea de notes em cima (padrão de hoje, ~2 rows iniciais).
- Lista de drafts logo abaixo com nome, tamanho, ícone por tipo,
  botão remover.
- Drop zone: aceita **paste de imagem** (⌘V), **drag-drop**, e
  **click pra abrir file picker**. Cada arquivo dropado vira um
  `AttachmentDraft` em memória até o submit do form.
- Validação síncrona por arquivo: MIME na whitelist do Spec 19
  (PNG/JPEG/WebP/PDF/CSV) + size < 10 MB. Se inválido, mostra erro
  inline e não adiciona o draft.
- Sem upload imediato — drafts ficam em memória até o submit do
  composer.

### 2. Integração nos 4 composers

`MovementComposer`, `DistributionComposer`, `TransactionComposer`,
`CardTxComposer`:

- Adicionar `const [files, setFiles] = useState<AttachmentDraft[]>([])`.
- Substituir a textarea atual por:

```tsx
<NotesAttachmentsField
  notes={notes}
  onNotesChange={setNotes}
  files={files}
  onFilesChange={setFiles}
  placeholder="ex: tese, motivo da compra, link pra notícia…"
/>
```

- No submit do composer: após `POST /asset-movements` (ou
  `/distributions`, etc.) retornar o `id` do novo registro,
  encadear `POST /attachments` para cada draft com
  `{ source_type, source_id, file }` (Spec 19). Em paralelo é OK.
- Tratamento de erro parcial: se 4 anexos e 1 falha, registro é
  salvo, anexos OK persistem, falha mostrada em toast com
  retry button.

### 3. Modo `editing` nos 4 composers

```typescript
interface MovementComposerProps {
  prefill?: Partial<MovementInput>
  editing?: boolean            // ← novo
  onClose: () => void
  onSaved?: (entity: AssetMovementOut) => void
}
```

Mudanças quando `editing=true`:

- `ModalHeader` title: "Novo Lançamento" → "Editar Lançamento"
  (e subtitle correspondente em cada composer).
- `ModalFooter` saveLabel: "Salvar" → "Salvar alterações".
- Submit usa `PUT /asset-movements/{id}` em vez de `POST`.
- `prefill` precisa carregar **todos** os campos da entidade,
  incluindo lista de anexos já persistidos (mostrar com botão
  remover funcional → DELETE soft do `Attachment` em vez de
  remover do array local).
- Anexos persistidos vs drafts: `<NotesAttachmentsField>` precisa
  diferenciar visualmente os 2 estados (persistido = ícone normal;
  draft = ícone com badge "novo" pequeno).

### 4. Botão Editar nos detail panels

`LancamentoDetailPanel` (Lançamento) e `DistributionDetailPanel`
(Provento):

```tsx
<button
  onClick={() => {
    composer.edit("movement", {
      id: m.id,
      type: m.movement_type,
      date: m.event_date,
      asset_id: m.asset_id,
      qty: m.quantity,
      price: m.unit_price,
      fee: m.fee,
      tax: m.tax,
      notes: m.notes,
      attachments: m.attachments,
    })
    onClose?.()                 // fecha o SidePanel
  }}
  className="...indigo-500 hover:indigo-400 text-white...">
  <Edit2 className="w-3.5 h-3.5"/> Editar
</button>
```

Posição: entre "Ver página do ativo" (esquerda) e "Inativar"
(direita).

Detail panels precisam de prop `onClose?: () => void` opcional —
parent (AssetDetail, LancamentosPage, ProventosPage) passa
`() => setSelected(null)`.

### 5. `composer.edit()` no ComposerProvider

`frontend/src/components/ComposerProvider.tsx`:

```typescript
interface ComposerAPI {
  open: (type: ComposerType, prefill?: any) => void
  edit: (type: ComposerType, prefill: any) => void   // ← novo
  close: () => void
}
```

`.edit()` é açúcar sintático para `.open(type, prefill, /*editing*/true)`.
Estado interno carrega `editing: boolean` que o host passa pros
composers.

### 6. AssetMovement/Distribution PUT endpoints

Se já existem `PUT /asset-movements/{id}` e `PUT /distributions/{id}`,
verificar que aceitam **todos** os campos do composer. Se faltarem
campos (ex: notes), estender o serializer.

Se **não** existem, criar como parte desta spec — payload espelha
o de POST.

### 7. Audit log

Cada edit gera entry com `action="movement.update"` (ou
`"distribution.update"`), `resource=<entity_type>/<id>`,
`details=<JSON diff: campo→{old, new}>`. Reusa pattern da Spec 03.

## Out of Scope

- **Editar Transaction / CardTransaction** via detail panel. Esses
  ainda não têm detail panel próprio no app real (vêm em spec
  futura). O **composer** ganha modo edit aqui pra quando for
  invocado externamente.
- **Bulk edit** (editar N lançamentos de uma vez).
- **Undo** após edit. Não tem; tem audit log + reabrir.
- **Versionamento de anexos** (substituir um anexo com upload novo
  mantendo histórico). Hoje é remove + add.
- **OCR/extração de anexos no momento do upload no composer**. Isso
  é Spec 38 (LLM extraction) e aplica só pra snapshot pendencies,
  não pra anexos de lançamento.
- **Anexos em `NewAssetComposer`, `NewAccountComposer`,
  `NewCardComposer`**. Cadastros raramente pedem comprovante;
  fora do escopo. Reabrir spec se virar pedido recorrente.

## Pre-implementation interview (Planner agent)

Decisões pra alinhar com o user antes de codar:

1. **Submit chain** — se 1 anexo falha após registro salvo, a
   transação como um todo é considerada parcial (mostrar toast)
   ou rollback do registro inteiro (mais consistente mas pior UX)?
   Sugestão: parcial com retry.
2. **Anexos persistidos no edit** — botão remover dispara DELETE
   imediato (Spec 19 soft delete) ou só marca pendente e remove
   no submit? Sugestão: imediato (mais simples, audit log cobre).
3. **Atalhos** — manter `⌘↵` salvar + `esc` fechar nos composers
   em modo edit também? Sugestão: sim, sem mudança.
4. **Confirmação de close** se há draft de anexos não salvos —
   mostrar "Tem certeza?" no esc/X? Sugestão: sim, padrão de
   forms com mudanças pendentes.
5. **PUT payload** — só campos mudados (PATCH-like) ou full
   replace (PUT clássico)? Sugestão: full replace pra simplicidade
   (compara no audit log diff).
6. **Audit log granularidade** — guardar diff de **todos** os
   campos ou só dos que mudaram? Sugestão: só os que mudaram.
7. **Editar tipo de movimento** (ex: BUY → SELL) deve ser
   permitido? Tem implicação em posições e cálculos. Sugestão:
   permitir mas avisar com diálogo "isso vai recalcular sua
   posição em X".
8. **Anexos em modo edit do composer** — quando user já tinha 2
   anexos persistidos e adiciona 1 draft, ele vê os 3 misturados.
   Ordem: persistidos primeiro, drafts depois? Sugestão: sim,
   com separador visual sutil.
9. **Audit log** entry — adicionar `attachment.add` /
   `attachment.remove` quando edit mexe em anexos, separado do
   `movement.update`? Sugestão: sim, granular fica mais útil pra
   auditoria.
